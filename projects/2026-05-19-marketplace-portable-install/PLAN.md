# PLAN: Marketplace portable install — split into per-family plugins

## Context

The `krambuhl/agents` repo (this marketplace) currently ships as a
monolithic install via `install.sh` (symlink farm into
`~/.agents/` + `~/.claude/{skills,agents}/`). The
research dossier at `RESEARCH.md` in this project establishes the
goal: install on any repo as a Claude Code plugin without coupling to
a personal clone, without enrolling colleagues, with griot learnings
in the consumer project.

This plan supersedes RESEARCH § A's tentative "one mega-plugin"
direction with a **granular per-family plugin shape** based on a
plan-session reframe: the marketplace's content has three conceptual
layers (substrate primitives, opinionated implementations, standalones)
that deserve independent installability. The lock-in cost is shared-
CLI-code duplication; the gain is consumer choice and clean conceptual
mapping.

**Research foundation**: `projects/2026-05-19-marketplace-portable-install/RESEARCH.md`
(committed earlier in this session). All RESEARCH § A-H working
directions still apply EXCEPT § A's "one mega-plugin" — explicitly
superseded here.

## Scope

### In

- Six plugin entries in `.claude-plugin/marketplace.json`:
  - `griot@krambuhl` — `griot-*` skills + `griot` CLI + `griot-*`
    agents (`griot-base`, `griot-judge`, `griot-rewriter`,
    `griot-operator`, `griot-rubric-author`, `griot-debate-summarizer`)
  - `guild@krambuhl` — `guild-*` skills + `guild` CLI +
    `whiteboard-*`, `evaluator-*`, `generator-*` agents
  - `loom@krambuhl` — `loom-*` skills + `loom` CLI. Declares
    dependency on `guild@krambuhl` (loom-research and loom-plan
    compose guild skills) and `griot@krambuhl` (startup-brief
    convention requires `bin/griot use --as=llm`).
  - `ev@krambuhl` — `ev-loop-confidence`, `ev-loop-interactive`,
    `ev-run` skills. Declares dependency on `loom`, `guild`, `griot`.
  - `review-skill@krambuhl` — the `review-skill` skill alone, no
    deps.
  - `agent-loop-full@krambuhl` — meta-plugin bundling all five (the
    turnkey install).
- New marketplace directory layout: `plugins/<name>/` per plugin,
  each self-contained. The authoritative top-level
  `cli/`/`skills/`/`agents/` becomes source-of-truth; per-plugin
  trees are generated by a sync script.
- New `scripts/sync-shared.ts` that copies the shared `cli/lib/`
  + cross-cutting verbs into each plugin's `cli/lib/`. Pre-commit
  hook (or CI check) fails if drift detected.
- New `bin/griot init` verb (Category 3 in `projects/CONVENTIONS.md`,
  new declared exception `gitignore-amendment`): idempotently
  scaffolds `./learnings/` in the consumer project and amends
  `.gitignore` to include `learnings/`.
- Griot CLI verbs change resolution from `ctx.cwd` to project-root
  walk-up using `.git/` as the marker; cwd fallback when no `.git/`
  is found. Affects `cli/verbs/griot/capture.ts:446`,
  `cli/verbs/griot/use.ts:11,129`, and `cli/verbs/griot/index.ts`'s
  `GriotCliContext`.
- Per-plugin `bin/<cli>` shims with Node version check
  (`>=24` required; emits actionable error otherwise).
- Skill body rewrites across all 5 plugins: every
  `Bash("bin/loom ...")` → `Bash("loom ...")`, etc.
- `ev/*` skill body preflight: `command -v loom guild griot >/dev/null || fail`
  at the top of each ev-* skill body. Fails loud with actionable
  message if any dep plugin isn't enabled.
- Manifest-level dependency declarations in marketplace.json (if
  Claude Code's API supports `requires` or equivalent — see
  Verification V4).
- Delete `install.sh` (no deprecation period; user is the only
  consumer).
- README rewrite. New install instructions: `claude plugin install
  agent-loop-full@krambuhl --scope local` (turnkey) or per-family
  `claude plugin install loom@krambuhl --scope local` (granular).
- Cleanup of stale references in skill bodies (the `~/.agents/...`
  paths flagged by RESEARCH Phase 0 V3 audit).

### Out (deferred or never)

- `bin/loom project untether` verb. The user hand-`rm`s shims from

- Orphan-skill prefix audit (the unprefixed `review-skill` and the

- Compile-to-JS bundling for the CLIs. RESEARCH § C settled on
  ship-source + Node 24 requirement.
- Loom-substrate refactor to split per family. The loom CLI stays
  monolithic inside `loom@krambuhl`; only the marketplace ships
  granularly.

## Phases

Per PQ1 (re-confirmed for the 5-plugin shape per PQ4 follow-up):
**one bundled migration PR**. No phase decomposition within the
migration itself.

The work splits into ordered workstreams inside the one PR, useful
for incremental local development but not as separate commits:

1. **Marketplace skeleton**: `.claude-plugin/marketplace.json` with
   6 entries; `plugins/<name>/` empty directories; sync script
   skeleton.
2. **Sync script implementation**: `scripts/sync-shared.ts` reads
   top-level `cli/lib/` and `cli/verbs/` and writes per-plugin
   subsets. Test it locally; commit the generated trees.
3. **Per-plugin bin shims**: write `plugins/<name>/bin/<cli>` with
   Node version check. Test each shim resolves correctly when
   invoked.
4. **`bin/griot init` verb**: new file `cli/verbs/griot/init.ts`,
   tests, dispatch wiring in `cli/verbs/griot/index.ts`. Sync
   into `plugins/griot/`.
5. **Griot project-root resolution**: refactor the `ctx.cwd` →
   walk-up logic in `cli/verbs/griot/`. Sync into the plugin.
6. **Skill body rewrites**: pass through every `SKILL.md` and
   replace `Bash("bin/<cli> ...")` with `Bash("<cli> ...")`.
   Audit for stale `~/.agents/...` references (V3) at the same time.
7. **`ev/*` preflight**: add `command -v` checks to each ev-loop-*
   and ev-run skill body.
8. **Documentation**: rewrite README and install docs to point at
   plugin install. Document the per-user `--scope local` pattern
   and the gitignore requirement for `.claude/settings.local.json`.
9. **install.sh deletion**: remove the file and remove all
   references from documentation.

## Dependencies

### Inter-plugin (within the marketplace)

- `loom@krambuhl` requires `guild@krambuhl` + `griot@krambuhl`.
  Declared in marketplace.json (V4) AND enforced via skill-body
  preflight.
- `ev@krambuhl` requires `loom@krambuhl` + `guild@krambuhl` +
  `griot@krambuhl`. Same belt-and-suspenders pattern.
- `griot@krambuhl`, `guild@krambuhl`, `review-skill@krambuhl`
  have no inter-plugin deps.
- `agent-loop-full@krambuhl` requires all five (its whole reason
  for existing).

### External

- Node 24+ runtime (declared in each plugin's bin shim).
- Bash (the bin shims are bash scripts).
- `gh` CLI is NOT a runtime dependency — only used in the
  framework's PR-handling verbs (`cli/verbs/pr/`).
- Claude Code itself, for plugin install + skill execution.

### Build-time

- `scripts/sync-shared.ts` runs at pre-commit + CI to keep
  `plugins/<name>/cli/lib/` in sync with `cli/lib/`. Failure mode:
  CI fails on drift; pre-commit catches before commit.

## Verification

### V1 — `.claude/settings.local.json` enrollment safety (from RESEARCH)

Empirical smoke test: commit a `.claude/settings.local.json`
containing the `enabledPlugins` entry into a sandbox repo. Clone
from a different user account / machine. Open in Claude Code.
Record behavior. **Pass criteria**: Claude Code does NOT silently
auto-install the plugin without user confirmation. **Failure
criteria**: silent auto-install — the dossier's "invisible to
colleagues" goal is unenforceable and we need a different scope.

### V2 — Node 24 on Patreon Bash tool (from RESEARCH)

Run `Bash("node --version")` from Claude Code inside a Patreon
repo. Confirm output reports Node >= 24. **Pass**: confirms the
shim's version check accepts the Patreon environment as-is.
**Failure**: identify the lowest Node version in active Patreon use
and revise the shim's minimum version or revisit the compile-to-JS
escape hatch.

### V3 — Skill-body `~/.agents/...` path audit (from RESEARCH)

`grep -rn '~/.agents/' skills/` (or, post-migration,
`plugins/*/skills/`). Triage each hit. **Pass**: zero hits, OR
each hit has an explicit rewrite to a plugin-install-compatible
resolution. **Failure**: hits that have no clean rewrite — those
skills will break after migration.

### V4 — Marketplace.json supports inter-plugin `requires` field

Verify whether Claude Code's marketplace.json schema includes a
`requires` (or equivalent) field on plugin entries that gates
install on dependencies. **Pass**: declare deps in manifest.
**Failure**: rely on runtime preflight + the meta-plugin pattern
(`agent-loop-full@krambuhl`) for users who want turnkey, and
document the manual-install order for granular users.

Read https://code.claude.com/docs/en/plugin-marketplaces.md +
https://code.claude.com/docs/en/plugins-reference.md +
spawn `claude-code-guide` to verify.

### V5 — Meta-plugin (`agent-loop-full`) implementation shape

If Claude Code supports `requires`, `agent-loop-full` is a
zero-content plugin that declares all five others as `requires`;
install triggers cascade-install. If `requires` isn't a thing,
`agent-loop-full` either (a) contains COPIES of everything
(defeats the granular split), or (b) is implemented as a
post-install script that runs five `claude plugin install`
commands, or (c) ships as documentation only ("to install the
turnkey, run these five commands").

Plan-implementation decision: spike each option once V4 is
verified, pick the cleanest.

### V6 — Full install smoke test on personal machine

After the migration PR is merged (repo still private or just-made-
public, depending on V8): run `claude plugin install
agent-loop-full@krambuhl --scope user` on personal Mac. Run a
small skill end-to-end (e.g. `/grill-me` on a sandbox topic, or
`/loom-research` on a dummy topic). **Pass**: skill runs without
error. **Failure**: identify the broken step (manifest, bin shim,
skill body, etc.) and fix before V7.

### V7 — Full install smoke test on Patreon machine

After V6 passes: repeat the install + smoke test on the Patreon
work machine. Same pass/fail criteria. **Additionally**: verify
that colleagues' sessions (their own Claude Code on the same repo)
do NOT show the plugin in their settings or available commands.
This is the load-bearing test of the "invisible to colleagues"
constraint (related to V1 but specific to the real Patreon
environment).

### V8 — Public-publish gating sequence

Open question for plan-implementation: does the repo need to be
public for `claude plugin install krambuhl/...` to work, or does
Claude Code support installing from private repos (e.g. via gh
auth)? Verify via Claude Code docs + claude-code-guide. Two
sequences depending on the answer:

- **If public required**: migration PR merge → make repo public →
  V6 → V7 → announce.
- **If private supported**: migration PR merge → V6 (private) → V7
  (private) → make repo public → announce.

The choice between V6-before-public vs V6-after-public is
operationally important — testing in private gives a safety
buffer for catching issues before public exposure.

## Risks

### R1 — Shared lib drift (high likelihood, medium impact)

`plugins/<name>/cli/lib/` drift from authoritative `cli/lib/`. The
sync script + pre-commit hook + CI check are the mitigation, but
a forgotten sync still ships if any of those layers fail.

**Mitigation**: hard-fail in CI on drift; treat the sync as a
required check on PRs. Consider deleting the per-plugin `cli/lib/`
from git entirely and regenerating at plugin-publish time only
(requires Claude Code's plugin install to include a build step;
unclear if supported).

### R2 — Inter-plugin install order without manifest deps (medium / high)

If V4 finds no manifest `requires` support, users who install
`ev@krambuhl` first will hit runtime preflight failures until they
also install loom/guild/griot. UX is rough.

**Mitigation**: the `agent-loop-full@krambuhl` meta-plugin solves
this for turnkey install. README + skill-body error messages need
to be crystal-clear about the dep chain for granular users.

### R3 — Patreon Node version drift over time (low / high)

Per RESEARCH § C, Patreon is currently on Node 24. If a future
Patreon repo pins an older Node via `.nvmrc`, the shim's version
check fails on every skill invocation. Loud failure, but
disruptive.

**Mitigation**: V2 verifies the current state. If Patreon ever
downgrades or pins older Nodes, revisit RESEARCH skeptic Finding 2
remedies (volta fallback, or the compile-to-JS escape).

### R4 — `.claude/settings.local.json` accidentally committed (medium / high)

Per RESEARCH skeptic Finding 4: if the gitignore isn't in place
and the user `git add .`s carelessly, colleagues get silently
enrolled.

**Mitigation**: V1 establishes the actual failure mode. The
plugin's install instructions explicitly require adding
`.claude/settings.local.json` to the consumer's `.gitignore`. The
user's global gitignore should also include it (defense in depth).
Optionally a monitor hook in the plugin warns on `git diff --cached`
showing the file.

### R5 — Meta-plugin without `requires` is awkward (medium / medium)

If V4 says no `requires` support, V5's spike for `agent-loop-full`
shapes how clean the turnkey install actually is. Worst case:
"install the turnkey" becomes "run these five commands" — barely a
turnkey at all.

**Mitigation**: V4 + V5 are sequenced first. If both come back
negative for clean meta-plugin support, revisit PQ-style decisions:
maybe one bundled mega-plugin really was right (the "revisit"
option from the plan-shape question).

### R6 — One-bundled-PR is large (medium / low)

The PR ships 6 plugin trees, marketplace.json, sync script, new
verb, refactor of griot resolution, rewrites across every skill
body. Reviewer cognitive load is high.

**Mitigation**: the user reviews their own PR — single-reviewer
cognitive load is bounded by the user's own attention. The
mechanical rewrites are easy to scan; the non-mechanical changes
(manifest, sync script, griot init verb, project-root resolution)
are concentrated in a handful of files reviewable closely.

## Open questions (for plan-implementation)

Things this plan did not fully resolve and the implementation phase
needs to decide:

- **OQ1**: Plugin source path convention in marketplace.json. The
  verified API shows `source` as a relative path string; exact
  resolution semantics (relative to marketplace root? relative to
  manifest?) need confirmation before authoring the manifest.
- **OQ2**: Whether `plugins/<name>/cli/lib/` is committed to git
  or `.gitignore`d (regenerated by sync). Committing makes
  every framework change a two-file diff; gitignoring requires the
  plugin publish path to run the sync.
- **OQ3**: Per-plugin manifest metadata (`plugin.json` inside each
  plugin source dir?) — what fields, what version pinning convention.
- **OQ4**: Whether `review-skill@krambuhl` deserves to live in this
  marketplace at all, or move to a personal `krambuhl/skills`
  marketplace. The dossier suggested keeping it in this marketplace
  for ease of finding; reconsider if the marketplace's scope
  feels muddied.
- **OQ5**: `bin/griot init` exact behavior: does it create
  `learnings/session-notes/`, `learnings/nightly/`, both, neither?
  What goes in the gitignore amendment block (`learnings/` alone, or
  also `learnings/!.gitkeep`)?

## Decisions (this plan's resolved interview)

See `INTERVIEW.md` for the full walked decision tree. Summary:

| Decision | Choice | Rationale |
|---|---|---|
| PQ1 sequencing | One bundled migration PR | Per user. Validates the granular split atomically. |
| PQ2 griot init timing | Ships in the migration PR | Verb is small; the invariant deserves enforcement. |
| Plugin shape | One plugin per family (5 plugins + meta) | Per user. Matches conceptual layering: substrate / impls / standalone. |
| review-skill placement | Own plugin `review-skill@krambuhl` | Family-of-one; installable without the framework. |
| a11y-review-file fate | Delete (pre-migration) | Per user. |
| Shared CLI code | Duplicate per plugin via sync script | Per user. Simpler than npm publishing; CI catches drift. |
| Inter-plugin deps | Manifest `requires` + runtime preflight + meta-plugin | Belt-and-suspenders; meta serves turnkey users. |
| install.sh fate | Delete in migration PR | Per user. No deprecation period; user is sole consumer. |
