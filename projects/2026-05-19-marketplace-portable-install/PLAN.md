# PLAN: Marketplace portable install — split into per-family plugins

## Context

The `krambuhl/agents` repo (this marketplace) currently ships as a
monolithic install via `install.sh` (symlink farm into
`~/.agents/` + `~/.claude/{skills,agents}/`). The
research dossier at `RESEARCH.md` in this project establishes the
goal: install on any repo as a Claude Code plugin without coupling to
a personal clone, without enrolling colleagues, with griot learnings
in the consumer project.

This plan supersedes RESEARCH § A's tentative "one mega-plugin"
direction with a **granular per-family plugin shape** based on a
plan-session reframe: the marketplace's content has three conceptual
layers (substrate primitives, opinionated implementations, standalones)
that deserve independent installability. The lock-in cost is shared-
CLI-code duplication; the gain is consumer choice and clean conceptual
mapping.

**Research foundation**: `projects/2026-05-19-marketplace-portable-install/RESEARCH.md`
(committed earlier in this session). All RESEARCH § A-H working
directions still apply EXCEPT § A's "one mega-plugin" — explicitly
superseded here.

**Whiteboard foundation**:
`projects/2026-05-19-marketplace-portable-install/whiteboards/1-execution-shape.md`
(Phase 1 round 1, pre-execution panel). Surfaced the workstream
re-ordering, the OQ2 resolution, the hidden pre-conditions in W6,
and the three new tests now baked into § Verification.

order below is the **substrate-shape order** (per the Phase 1

## Dependencies

### Inter-plugin (within the marketplace)

- `loom@krambuhl` requires `guild@krambuhl` + `griot@krambuhl`.
  Declared in marketplace.json (V4) AND enforced via skill-body
  preflight.
- `ev@krambuhl` requires `loom@krambuhl` + `guild@krambuhl` +
  `griot@krambuhl`. Same belt-and-suspenders pattern.
- `griot@krambuhl`, `guild@krambuhl`, `review-skill@krambuhl`
  have no inter-plugin deps.
- `agent-loop-full@krambuhl` requires all five (its whole reason
  for existing).

### External

- Node 24+ runtime (declared in each plugin's bin shim).
- Bash (the bin shims are bash scripts).
- `gh` CLI is NOT a runtime dependency — only used in the
  framework's PR-handling verbs (`cli/verbs/pr/`).
- Claude Code itself, for plugin install + skill execution.

### Build-time

- `scripts/sync-shared.ts` runs at pre-commit + CI to keep
  `plugins/<name>/cli/lib/` in sync with `cli/lib/`. Failure mode:
  CI fails on drift; pre-commit catches before commit.

## Verification

### V1 — `.claude/settings.local.json` enrollment safety (from RESEARCH)

Empirical smoke test: commit a `.claude/settings.local.json`
containing the `enabledPlugins` entry into a sandbox repo. Clone
from a different user account / machine. Open in Claude Code.
Record behavior. **Pass criteria**: Claude Code does NOT silently
auto-install the plugin without user confirmation. **Failure
criteria**: silent auto-install — the dossier's "invisible to
colleagues" goal is unenforceable and we need a different scope.
**Gating**: runs in W0 before the meta-plugin scope decision; the
README's install instructions depend on its outcome.

### V2 — Node 24 on Patreon Bash tool (from RESEARCH)

Run `Bash("node --version")` from Claude Code inside a Patreon
repo. Confirm output reports Node >= 24. **Pass**: confirms the
shim's version check accepts the Patreon environment as-is.
**Failure**: identify the lowest Node version in active Patreon use
and revise the shim's minimum version or revisit the compile-to-JS
escape hatch.

### V3 — Skill-body `~/.agents/...` path audit (from RESEARCH)

`grep -rn '~/.agents/' skills/` (or, post-migration,
`plugins/*/skills/`). Triage each hit. **Pass**: zero hits, OR
each hit has an explicit rewrite to a plugin-install-compatible
resolution. **Failure**: hits that have no clean rewrite — those
skills will break after migration. **Pre-W6 design call already
made**: option (c) — inline or `bin/<cli> docs` verb invocation
(see § Scope).

### V4 — Marketplace.json supports inter-plugin `requires` field

Verify whether Claude Code's marketplace.json schema includes a
`requires` (or equivalent) field on plugin entries that gates
install on dependencies. **Pass**: declare deps in manifest.
**Failure**: rely on runtime preflight + the meta-plugin pattern
(`agent-loop-full@krambuhl`) for users who want turnkey, and
document the manual-install order for granular users. **Gating**:
runs in W0 before manifest authoring; if V4 fails, pivot the
meta-plugin to documentation-only (option (c) per V5).

Read https://code.claude.com/docs/en/plugin-marketplaces.md +
https://code.claude.com/docs/en/plugins-reference.md +
spawn `claude-code-guide` to verify.

### V5 — Meta-plugin (`agent-loop-full`) implementation shape

If V4 confirms `requires`, `agent-loop-full` is a zero-content
plugin that declares all five others as `requires`; install
triggers cascade-install. If `requires` isn't supported, the
substrate-fit answer is option (c) — ship as documentation only
("to install the turnkey, run these five commands") — because
option (a) doubles drift surface and option (b) makes the meta-
plugin a different shape from its siblings (per substrate-engineer
Finding 6).

Plan-implementation decision: V5 spike runs in W0 alongside V4;
the outcome is captured in the PR body so the reviewer ratifies
the chosen shape against an in-diff fact.

### V6 — Full install smoke test on personal machine

After the migration PR is merged (repo still private or just-made-
public, depending on V8): run `claude plugin install
agent-loop-full@krambuhl --scope user` on personal Mac. Run a
small skill end-to-end (e.g. `/grill-me` on a sandbox topic, or
`/loom-research` on a dummy topic). **Pass**: skill runs without
error. **Failure**: identify the broken step (manifest, bin shim,
skill body, etc.) and fix before V7. **Formalize as a written
checklist** (per testing-strategy panel): the smoke test is the
e2e tier for this entire migration; without a reproducible
artifact it's a one-time human run, not a test.

### V7 — Full install smoke test on Patreon machine

After V6 passes: repeat the install + smoke test on the Patreon
work machine. Same pass/fail criteria. **Additionally**: verify
that colleagues' sessions (their own Claude Code on the same repo)
do NOT show the plugin in their settings or available commands.
This is the load-bearing test of the "invisible to colleagues"
constraint (related to V1 but specific to the real Patreon
environment). **Formalize as a written checklist** alongside V6.

### V8 — Public-publish gating sequence

Open question for plan-implementation: does the repo need to be
public for `claude plugin install krambuhl/...` to work, or does
Claude Code support installing from private repos (e.g. via gh
auth)? Verify via Claude Code docs + claude-code-guide. Two
sequences depending on the answer:

- **If public required**: migration PR merge → make repo public →
  V6 → V7 → announce.
- **If private supported**: migration PR merge → V6 (private) → V7
  (private) → make repo public → announce.

The choice between V6-before-public vs V6-after-public is
operationally important — testing in private gives a safety
buffer for catching issues before public exposure.

### V9 — `marketplace-manifest.test.ts` (new, registry-vs-reality tripwire)

Unit test that reads `.claude-plugin/marketplace.json`, asserts the
six plugin entries exist with required `name`/`source` fields, and
asserts each declared `source` points at a directory that actually
exists on disk. Same flavor as `parallel-work-invariant.test.ts`.
Catches the "manifest is malformed and `claude plugin install`
rejects it silently" failure class. **Lands in W1** alongside the
manifest itself.

### V10 — `sync-shared.test.ts` (new, drift-detection tripwire)

Three test cases: (a) writes correctly given a source tree and
target tree; (b) end-to-end fixture in tmpdir, per-plugin
`cli/lib/` matches authoritative `cli/lib/` byte-for-byte;
(c) **drift detection** — mutates a per-plugin file post-sync and
asserts the CI-check exits non-zero. Case (c) catches the
false-green failure mode (R1's core risk). **Lands in W7**
alongside `scripts/sync-shared.ts`.

### V11 — `skill-bodies-call-bare-commands.test.ts` (new, grep tripwire)

Single repo-wide grep test: `find plugins/*/skills/ -name '*.md'
-print0 | xargs grep -l 'Bash("bin/'` returns zero hits, OR the
test asserts that. Defends against missed shim references in the
bulk skill-body rewrite. **Lands in W4** alongside the rewrite.

## Risks

### R1 — Shared lib drift (high likelihood, medium impact)

`plugins/<name>/cli/lib/` drift from authoritative `cli/lib/`. The
sync script + pre-commit hook + CI check are the mitigation, but
a forgotten sync still ships if any of those layers fail.

**Mitigation**: hard-fail in CI on drift (V10 case (c)); treat the
sync as a required check on PRs. Decision: **commit the generated
trees** (per § Decisions OQ2 resolution); drift becomes a visible
failure mode in CI rather than a silent plugin-publish-time
divergence.

### R2 — Inter-plugin install order without manifest deps (medium / high)

If V4 finds no manifest `requires` support, users who install
`ev@krambuhl` first will hit runtime preflight failures until they
also install loom/guild/griot. UX is rough.

**Mitigation**: the `agent-loop-full@krambuhl` meta-plugin (or
its docs-only equivalent per V5) solves this for turnkey install.
README + skill-body error messages need to be crystal-clear about
the dep chain for granular users.

### R3 — Patreon Node version drift over time (low / high)

Per RESEARCH § C, Patreon is currently on Node 24. If a future
Patreon repo pins an older Node via `.nvmrc`, the shim's version
check fails on every skill invocation. Loud failure, but
disruptive.

**Mitigation**: V2 verifies the current state. If Patreon ever
downgrades or pins older Nodes, revisit RESEARCH skeptic Finding 2
remedies (volta fallback, or the compile-to-JS escape).

### R4 — `.claude/settings.local.json` accidentally committed (medium / high)

Per RESEARCH skeptic Finding 4: if the gitignore isn't in place
and the user `git add .`s carelessly, colleagues get silently
enrolled.

**Mitigation**: V1 (in W0) establishes the actual failure mode.
The plugin's install instructions explicitly require adding
`.claude/settings.local.json` to the consumer's `.gitignore`. The
user's global gitignore should also include it (defense in depth).
Optionally a monitor hook in the plugin warns on `git diff --cached`
showing the file.

### R5 — Meta-plugin without `requires` is awkward (medium / medium)

If V4 says no `requires` support, V5's spike for `agent-loop-full`
shapes how clean the turnkey install actually is. Worst case:
"install the turnkey" becomes "run these five commands" — barely a
turnkey at all.

**Mitigation**: V4 + V5 are sequenced in W0. Substrate-fit answer
on failure: option (c) docs-only (not option (b) post-install
script). If both come back negative for clean meta-plugin support,
revisit PQ-style decisions: maybe one bundled mega-plugin really
was right (the "revisit" option from the plan-shape question).

### R6 — One-bundled-PR is large (medium / low)

The PR ships 6 plugin trees, marketplace.json, sync script, new
verb, refactor of griot resolution, rewrites across every skill
body. Reviewer cognitive load is high.

**Mitigation**: the user reviews their own PR — single-reviewer
cognitive load is bounded by the user's own attention. With OQ2
resolved (commit generated trees), the diff will be larger than
the alternative, but the mechanical sync output reviews fast
(deterministic, grep-able). The non-mechanical changes (manifest,
sync script, griot init verb, project-root resolution) are
concentrated in a handful of files reviewable closely. **PR body
explicitly names which files are sync-script output** so the
reviewer skips them confidently.

### R7 — Review-time duplication makes the diff hard to read (new, medium / medium)

Per skeptic Finding 2: committing generated trees means every
conceptual change shows up 2-6 times in the diff (authoritative +
per-plugin copies). The user reviews with high context now, but
future-self reading `git log` of this commit sees a 5,000-line
diff and can't tell which lines were intentional.

**Mitigation**: the PR description explicitly calls out which
files are sync-script output ("generated by `scripts/sync-shared.ts`,
do not review line-by-line; review the script + the source
instead"). Same posture as the existing `checkin-already-exists`
loud-fail: make review boundaries explicit in the diff itself.
