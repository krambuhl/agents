# Whiteboard: DESIGN.md composition for linear-loom: Linear-driven flow router for agent orchestration

## Round 1

### From whiteboard-substrate-engineer

This is a substrate plugin with a split state store — Linear is one source of truth, git is another, and the polling worker reconciles between them. That's a substantively different shape from how loom/guild/ev/griot are built today (each owns a single state root). The DESIGN.md needs to make that shape legible, not just functional. Here are the load-bearing decisions where the substrate's invariants are most at risk:

**1. Name and locate the source of truth for every piece of state, explicitly.** The decision-vs-process axis from RESEARCH.md (PLAN.md and ADR log in git; events/status/check-ins in Linear) is the right starting point, but it's a *category* split, not yet an *artifact* split. The DESIGN.md needs a table — one row per artifact (issue status, dispatch cursor, machine-state ledger, agent check-in, PR linkage, retro, whiteboard panel output, adapter config, idempotency key), each row naming exactly one canonical writer. Two writers to the same field is the substrate smell that causes every "Linear and git disagree" bug six months from now. Write the table; let it be the spec.

**2. The dispatch cursor is the substrate's hardest object — pick a shape with deliberate trade-offs.** PLAN.md's open question 2 names three options (Linear custom field on the issue, repo state file, Coder-side ledger). These are not equivalent under parallelism. A custom field on the issue is append-only per-issue and survives polling-run loss, but loses workspace-level "what was the last cursor" without a scan. A repo state file is workspace-scoped but introduces a git write from a polling worker (who commits it? what branch? what happens when two scheduled runs overlap and both try to push?). A Coder-side ledger requires a long-lived store outside Coder's ephemeral model, which contradicts the "ephemeral machines" framing. My lean: per-issue claim field on the Linear issue (idempotent by construction — setting it twice with the same dispatch ID is a no-op), plus an in-memory "last successful poll timestamp" the next run reads from Linear's own activity feed rather than from any external ledger. No git writes from the worker. State stays in Linear; git stays a read-only target for the worker.

**3. Idempotency is the orchestration contract — make it constructive, not vigilant.** The polling loop will fire at least twice on every transition during normal operation (the 5-minute cron overlapping the dispatch window) and arbitrarily many times during retries, drift, and re-runs. The contract must be: "dispatching the same issue's same transition N times produces exactly one machine spin-up." The cheapest way is a deterministic dispatch key (`issue-id + workflow-state-id + transition-timestamp`) written to the Linear issue *before* the Coder spin-up call, with a check-then-set semantics that uses Linear's update mutation as the lock. If Linear doesn't support compare-and-swap on a custom field (worth verifying — the docs surveyed in RESEARCH.md don't explicitly say), the design needs a fallback: either accept best-effort idempotency with a "duplicate dispatch detected" reconciler, or move the claim to a substrate that does (a GitHub Actions concurrency group keyed on issue ID is a viable shape and costs nothing). Press on the primitive; don't paper over it with "the polling worker checks twice."

**4. Adapter contract is the family-shape decision that compounds the most.** PLAN.md risks list this ("Agent flow adapter divergence") and rightly. The five flows named (`planning`, `iterative-confidence`, `iterative-interactive`, `whiteboarding`, `evaluator-panel`) compose five existing skill families. The adapter contract — what inputs each adapter takes from Linear, what outputs it writes back, what success/failure looks like — is the family shape every future flow joins. Define it once, in DESIGN.md, as a concrete TypeScript-ish interface with named fields and lifecycle hooks (`hydrate(issue) → context`, `run(context) → result`, `report(result) → linear-mutations[]`, `cleanup(machine) → void`). Without it, flow #6 lands six months from now and invents its own shape, and now you have a `linear-loom-*` family with two parallel conventions and no migration path. Same instinct as design-systems' "vocabulary consistency": cohesion compounds, parallel naming costs forever.

**5. Schema evolution: pre-decide what's additive and what requires migration.** Linear custom fields and label taxonomies are the substrate's schema. The DESIGN.md should declare an explicit schema_version on the linear-loom Linear configuration (a Document at a well-known title, or a custom field on the project itself), and a versioning policy: new flow types are additive (new label values, optional new custom fields); renames or shape-changes bump the version and require a one-time migration verb. Without this, a year from now someone renames a label and silently breaks every dispatched issue in flight. The cost of declaring the policy upfront is one paragraph; the cost of not declaring it is unbounded.

**6. CRUD-vs-orchestration boundary inside the linear-loom CLI.** PLAN.md lists "every linear-loom verb" as a DESIGN.md deliverable. The split worth pre-committing to: `bin/linear-loom` verbs are CRUD primitives against Linear (`issue update --status`, `dispatch claim --issue-id=`, `document write --slug=`), each a single Linear mutation with deterministic idempotency. Higher-level composition (the polling loop's full work cycle, "open a PR and update the Linear issue together," the agent-flow lifecycle) lives in `scripts/` or in the GitHub Actions workflow body or in skill files, calling the CLI primitives. The smell to avoid: a `bin/linear-loom poll` verb that runs the entire orchestration loop end-to-end and knows about every flow type. That verb has no single-responsibility shape, can't be tested in isolation, and breaks the "loom does CRUD, scripts do composition" pattern the existing plugin family uses. Push the orchestration up; keep the CLI surface flat and primitive.

**7. Substrate-gap surfacing.** A real one I'd want named in DESIGN.md: today there's no `bin/loom phase add` verb (per the user's memory note about loom gaps); linear-loom will hit equivalent gaps as it goes. The DESIGN.md should commit to a discipline — when the implementation phases discover that a needed verb doesn't exist (in linear-loom or in a sibling plugin), the gap is recorded as a first-class substrate artifact (a Linear issue tagged `substrate-gap`, or a sibling git project) rather than worked around in the agent body. Otherwise the workarounds calcify and the substrate never converges.

**Defer to siblings:** Cost-of-ephemeral-machines and polling-cadence math are `whiteboard-performance`'s lead — I'd point them at the 3M-points/hour budget and the N-agents-times-M-polls arithmetic in RESEARCH.md and let them price the steady state. Security/secret-distribution shape is its own lane (testing-strategy or a security-shaped evaluator if one exists — PLAN.md notes that gap explicitly as a Phase 1 sub-decision). I'm staying on shape.

**What I'd want resolved by round 2:** Which of the three dispatch-cursor shapes (issue-field, repo-state, Coder-ledger) the design is committing to, with the idempotency primitive named concretely. That decision cascades into half the failure-modes section, and leaving it as an open question through to implementation is the substrate risk I'd press hardest on.

### From whiteboard-performance

I'll engage selectively. Linear-loom is server-side orchestration — there's no React tree, no client bundle, no hydration story. The bundle-size and `'use client'` concerns my role usually presses on don't apply. But "performance" has a wider read here: **cost-per-poll, cost-per-dispatch, cost-per-machine-minute, and end-to-end latency from operator-action to first-token**. Those have the same shape as bundle/hydration concerns — small decisions at design time that compound at runtime — so I'll press there.

Six things I'd push the DESIGN.md author to address:

**1. Price the polling query before picking a cadence.** RESEARCH.md cites Linear's 3M complexity points/hour ceiling with 10K per single query. PLAN.md's risks section names this but doesn't yet quantify. The DESIGN.md should do the arithmetic explicitly — *here is the exact GraphQL query, here are the fields and nested edges, here is the complexity score per poll, here is the per-hour total at 5-minute cadence, here is the headroom for swarm scale.* A query that pulls issues with `assignee { teamMemberships { team { ... } } }` nested out two levels costs an order of magnitude more than one that pulls a flat id-plus-state. The seam to leave: keep the polling query *minimal* (id, updatedAt, the one custom field encoding flow type, the workflow-state id) and resolve the rest lazily inside the dispatched machine where it doesn't count against the polling budget. The poll is just "what's new and what flow does it want" — anything richer is the agent's job.

**2. Make the cursor model O(1)-shaped, not O(n)-shaped.** PLAN.md's open questions list three cursor options: Linear custom field on issue, state file in repo, Coder-side ledger. The performance shape of each is materially different. A per-issue custom field means *every poll fetches that field for every candidate issue and writes it back on dispatch* — that's two query-points per issue per cycle, scaling linearly with backlog. A single repo state file (or a single Linear Document with a JSON blob) is O(1) regardless of backlog size: one read at poll start, one write at poll end, no per-issue ceremony. The repo state file also has the property that `gh actions` already has it locally with no extra fetch. I'd push toward the repo-side cursor as default with explicit naming of when the per-issue field would be worth the cost (e.g. if multi-workflow racing requires per-issue claim semantics — but PLAN.md defers multi-operator coordination, so v1 probably doesn't need it).

**3. Cold-start latency is the real user-perceived latency, not poll cadence.** PLAN.md frames the 5-15 minute window as "polling latency," but that's the *floor*, not the ceiling. Operator-perceived latency from "drag to In Progress" to "agent says something useful in the issue comment" is: poll-wait (0-15 min) + Coder spin-up (?) + auth distribution (?) + plugin install / git clone / npm install (?) + skill warm-up (?) + first agent inference (?). The DESIGN.md should put a budget on each segment and identify which dominates. If Coder cold-start is 90 seconds and `npm install` is 3 minutes inside the machine, the poll-cadence debate is academic — the binding constraint is the spin-up tail. Mitigations worth comparing: a warm-pool of pre-baked machines, a base image with substrate plugins pre-installed, lazy clone vs. full clone, deferring `npm install` until after the first Linear-comment "starting work" acknowledgment lands (so the operator at least sees signal fast even if the agent isn't producing tokens yet).

**4. Idempotency keys and the cost of redundant dispatch.** PLAN.md's risks section mentions idempotency but doesn't yet shape it. The performance cost of a duplicate dispatch isn't just "two machines running" — it's two machines that both pull tokens from the Anthropic API, both write to Linear, and may produce conflicting PRs. The idempotency key should be cheap to compute, cheap to check, and live somewhere the dispatch decision can read in O(1). Concrete shape worth proposing: key = `{issue_id}:{workflow_state_id}:{flow_type}:{transitioned_at_iso}`. The check happens before machine spin-up — a duplicate key means "do nothing." Where the key lives is the design question: same place as the cursor, probably.

**5. Coder machine cost is a runtime hot path the design needs to budget at design time.** PLAN.md's risks section names "hard budget cap, max-machine-age timer, external reaper." Push that further: the DESIGN.md should name *expected steady-state cost per dispatched task* (machine-minutes × per-minute rate + Anthropic tokens × per-token rate + Linear API points × $0 but with a ceiling). A dogfood project running 5 tasks/day at 30 machine-minutes each is one cost profile; a swarm running 50 tasks/day at 4 machine-hours each is another. The reaper logic should specify both *normal teardown* (flow signals "I'm done") and *abnormal teardown* (max age hit, budget exhausted, parent flow died, network partitioned mid-run). Specifically pin: what happens to the in-flight work when the reaper fires mid-flow? Does the agent get a SIGTERM grace window to commit/comment back to Linear, or is it a hard kill that loses state?

**6. Linear API "points" is one budget; Anthropic "tokens" is another; they can interact badly.** A flow that's chatty with Linear (status comments every few minutes, fine-grained progress updates) burns Linear points. A flow that's chatty with itself (lots of agent inference loops) burns Anthropic tokens. The two budgets are independent but the design choices that affect them aren't — a "verbose progress reporting" decision in an adapter contract increases both. The DESIGN.md adapter contract (Section 2 in your brief) should specify a *reporting cadence* — e.g. "adapters MUST comment on the Linear issue at start, at PR-open, and at end; SHOULD NOT comment more than once per N minutes for in-progress signals" — so per-flow chattiness doesn't escape into runtime cost. Cheapest thing to do at design time, most expensive thing to discover after dogfood.

**One thing I'm explicitly not pressing on**: the storage-axis decisions (PLAN.md / events / retros / decisions in git vs Linear). That's a contract / read-pattern question the React-architect and skeptic are better-positioned for, and the performance cost there is dominated by the *agent-read latency per artifact*, which is already named in RESEARCH.md's "unit contracts in Linear adds an MCP round-trip per unit-of-work" caveat. I'd just second that caveat: any artifact agents read on a per-unit-of-work hot path should default to git (no round-trip); artifacts agents read once-per-project can default to Linear (one round-trip amortized).

### From whiteboard-skeptic

I do engage — this brief is wall-to-wall load-bearing decisions with high reversibility cost, which is exactly the territory the pressure-test lens is built for. Five concerns, ordered by how expensive they are to fix later.

**1. The "Linear custom field as cursor" pattern hides a multi-writer consistency problem you have not priced.**

PLAN.md names three candidate cursor locations — a Linear custom field, a state file in the repo, a Coder-side ledger — and treats the choice as a coin-flip. It is not. Each choice has a different consistency model and a different failure mode under the very race you've already flagged in the Risks section:

- **Linear custom field as cursor**: the cursor *is* the source-of-truth state, and the dispatcher modifies it via the same API it polls. Risk: read-modify-write across two GraphQL calls is not atomic. Two GitHub Actions runs that fire in overlapping windows (which happens routinely under cron drift) will both read "not dispatched," both spin up machines, both write "dispatched." You need either Linear's optimistic-concurrency token (if their API exposes one — verify before designing around it) or a separate claim mechanism (assignee swap as the atomic primitive, with the cursor merely descriptive).
- **State file in the repo**: now your cursor lives in git, which means the polling workflow has to `git commit && git push` every run, which means every run produces a commit on `main` (or a state branch), which means your repo's commit log is now half cursor noise. It also means two parallel GH Actions runs race on `git push` — one wins, one rejects, and you're back to needing application-level retry logic.
- **Coder-side ledger**: introduces a hard dependency on the ephemeral substrate being durable, which contradicts "ephemeral." If the ledger has to outlive the machine, it isn't Coder-side; it's some other store you haven't named.

The DESIGN.md author should pick one and **show the race trace** — two concurrent polling runs, what each reads, what each writes, where the conflict resolves. Not as belt-and-suspenders prose; as a worked example with explicit ordering. If you can't write the trace, the design isn't ready.

**Concrete remedy**: lean toward **assignee-as-claim** with custom-field-as-descriptive-cursor. The assignee field is the only Linear primitive with a server-side compare-and-swap shape (the API rejects an assignment that conflicts with a concurrent edit, if you use the right mutation). The cursor becomes informational, not authoritative.

**2. Polling cadence design is treating "5-minute cron" as the design parameter when it's actually the floor.**

RESEARCH.md prices the 3M points/hour ceiling correctly. PLAN.md leans on "narrow filters" as the mitigation. Both are correct individually and insufficient together. The thing nobody has priced: the query budget is not just polling — it's polling + every dispatched agent doing its own Linear reads (the agent flows comment back, update status, fetch context, write Documents, etc.) on the same workspace token unless you isolate auth per machine. At swarm scale (even a modest one — say 8 concurrent agents) you're sharing the 3M ceiling across the orchestrator AND the agents, and your polling budget shrinks accordingly.

The DESIGN.md author should write out the **per-hour point budget allocation**:

- Polling baseline: 12 runs/hour × query cost (estimate using Linear's complexity formula, not vibes) = X points
- Per-agent steady-state: estimate per-flow Linear API usage (status updates, comments, document reads) × N concurrent agents = Y points
- Headroom: 30%+ for spikes
- Total: X + Y + headroom ≤ 3M

Without this arithmetic in DESIGN.md, "narrow filters" is a hope, not a design. If the arithmetic doesn't fit, the design needs to either (a) per-agent API keys with separate budgets, or (b) a coalescing layer that batches agent writes.

**3. The "ephemeral machine + auth distribution" model is the single most expensive thing to get wrong, and the current framing under-prices the secrets-rotation problem.**

PLAN.md names auth distribution as a concern. It frames the question as "scoping and expiration." It misses the harder question: **how do you revoke a leaked credential in a system where machines come and go on a 5-minute cadence and you don't know which machine got the bad copy?**

Concrete scenarios the design needs to handle:

- An agent flow throws an exception and dumps the env to logs. Logs ship to GH Actions, which retains them for 90 days, which is visible to anyone with repo read access.
- A machine fails to tear down (the "external reaper" case). The credential sits in `~/.config/somewhere` on a zombie VM for hours/days. You don't know which credential, on which machine.
- The Anthropic API key is broadly-scoped (typical), so a single leak rotates everything for all in-flight work.

**Concrete remedy**: design around **per-task short-lived credentials**, not long-lived workspace credentials. Specifically:
- Linear: per-task OAuth token issued at dispatch with TTL ≤ task expected duration + buffer, scoped to the issue's project.
- GitHub: ephemeral GitHub App installation token (1-hour TTL is the platform default; renewable from inside the machine with the App's signing key, which itself never leaves the orchestrator).
- Anthropic: this is the hardest because Anthropic API keys are not naturally scoped per-task. The honest answer is probably "one workspace key, rotated weekly, with a per-task usage budget enforced upstream of the API call." DESIGN.md should *say that* explicitly rather than imply per-task isolation that isn't actually achievable.

The point is not that you'll get all of this right in v1. The point is that DESIGN.md should **name which credential leaks are recoverable in minutes vs hours vs require workspace-wide rotation**, so the operator running this isn't surprised the first time something goes wrong.

**4. The adapter contract is the API-shape decision that gets locked in by the second consumer, and the plan defers it to "phase 5.x."**

PLAN.md sketches five flow types (`planning`, `iterative-confidence`, `iterative-interactive`, `whiteboarding`, `evaluator-panel`) and says each gets its own adapter, with the adapter contract to be defined in DESIGN.md. Good — but the order matters. The contract has to be designed against **the hardest flow first**, not the easiest. If the contract is shaped by `planning` (which is essentially a one-shot stateless invocation) and then `evaluator-panel` (multi-round, parallel sub-agents, structured panel output, blocking-vs-advisory verdicts) tries to fit into that shape, you'll either bend the contract or special-case the panel flow. Both compound.

**Concrete remedy**: DESIGN.md's adapter-contract section should walk through `evaluator-panel` and `iterative-interactive` first as the contract-shaping cases, then verify the simpler flows fit. The contract needs at minimum:
- Multi-round / single-round axis (some flows finalize on round 1; some don't)
- Sub-dispatch capability (can a flow spawn other flows on its machine, or does it bounce work back to Linear and let the orchestrator re-dispatch?)
- Verdict shape (success/fail isn't enough; `guild-validate` returns a structured panel verdict with blocking/advisory findings)
- Cancellation semantics (operator drags issue out of dispatchable state mid-flow — what happens to the running machine?)

The cancellation case is the one I'd most expect to get missed in round 1 and bite you in dogfood.

**5. There is no design for the "linear-loom is broken and the operator needs to manually intervene" path.**

This is the meta-concern that probably matters most. Every section of PLAN.md assumes the happy path: cron fires, polling reads, dispatch succeeds, agent runs, agent reports back, PR merges, Linear flips to Done. None of the sections describe what the operator *does* when any step fails. Concretely:

- Issue stuck in "In Progress" with no associated Coder machine — how does the operator tell? What button do they push to retry, or to give up and reset the cursor?
- Coder machine running but agent flow hung — how does the operator see this? Where do the logs go?
- Polling workflow silently failing (GH Actions disabled the schedule because the repo went 60 days without push activity, which is a real GitHub behavior) — how does the operator notice?

**Concrete remedy**: DESIGN.md needs an **operator runbook section** alongside the architecture sections. Not deployment instructions — a "here's what you do when X is wrong" decision tree. The CLI surface should include diagnostic and manual-override verbs as first-class citizens, not afterthoughts:
- `linear-loom status <issue-id>` — show dispatch state, machine state, agent state
- `linear-loom reset <issue-id>` — clear the cursor, allow re-dispatch
- `linear-loom logs <issue-id>` — stream the agent flow logs from the (still-live or torn-down) machine
- `linear-loom poll --once --dry-run` — show what the next polling run would dispatch, without dispatching

Without these, your first dogfood failure becomes a multi-hour debugging session against an opaque distributed system. With them, it becomes a 30-second command.

**One thing I'd flag for round 2**: I'm withholding pressure on Coder CLI specifically until the other panelists weigh in. If `whiteboard-substrate-engineer` lands on Coder, I'll pressure-test the lock-in cost; if they propose a comparison matrix, I'll pressure-test the criteria. Right now my read is that the "Coder CLI or equivalent" framing in PLAN.md is doing too much work hiding a decision that probably wants to be made explicitly in DESIGN.md, but that's substrate-engineer's domain to lead.

### From whiteboard-testing-strategy

I'll engage. Testing strategy applies — not the obvious "what Vitest tests should the CLI have" question, but the deeper one: **how does anyone get fast, trustworthy signal that this thing is working when half the system lives on someone else's servers, the dispatch interval is 5-15 minutes, and the failure modes are the kind that compound silently?** That's a test architecture question, and it has to be answered in DESIGN.md or the implementer will reinvent it ad hoc per surface.

Six things I'd push you to address:

**1. Name the test tiers explicitly, and assign each behavior to exactly one.** Linear-loom has at least four natural tiers and the design should call them out so the implementation phases don't conflate them:

- **Pure-logic units** — query construction, idempotency-key derivation, cursor advancement, flow-type → adapter resolution, the label-vs-custom-field encoding/decoding. These should be pure functions. No Linear, no Coder, no time. Vitest, milliseconds.
- **Adapter contract tests** — each agent-flow adapter conforms to a shared shape (per PLAN.md risk: "without a shared adapter contract, each adapter risks drifting"). The contract itself deserves a test suite that every adapter runs against. Same shape as a TypeScript interface test, but for runtime I/O behavior.
- **Integration against a fake Linear** — the polling loop, dispatch logic, race semantics, claim protocol. These need a Linear-shaped fake that responds to the GraphQL subset linear-loom actually uses, not a full Linear simulator. The point is exercising the *boundary* (query shape, pagination, complexity-points-aware retry, "issue already claimed" race) without burning 3M points/hour during CI.
- **Smoke against a real sandbox Linear workspace** — a small set of "does this actually work end-to-end" tests against a dedicated Linear sandbox project. Run rarely (pre-release, post-deploy), not on every PR. This is where you catch "Linear's MCP behavior diverged from the docs we wrote against."

The risk DESIGN.md should price: if every adapter is tested only via the smoke tier because the fake doesn't exist, CI is slow, flaky, and expensive in Linear API points. If every adapter is tested only against pure mocks, the contract drifts from real Linear behavior and the dogfood phase finds it instead of CI. The fake-Linear layer is the one most likely to get skipped under time pressure, and it's the one that pays the most dividends. **Worth specifying in DESIGN.md what the fake covers, what it doesn't, and what the discipline is for keeping it honest** (e.g. record-replay from real Linear traffic, or a hand-maintained protocol fixture).

**2. The polling loop is the highest-risk surface and needs a deterministic test harness, not "let it run and see."** The risks PLAN.md already names (missed cycle, schedule drift, race conditions, idempotency) are all *temporal* — they happen because two things interleaved in a way you didn't expect. The cheapest defense is making time injectable: every call to `now()` and every cron tick goes through a clock the test can advance manually. Then you can write tests like:

- "two polling runs fire 30 seconds apart, both see the same issue in 'In Progress', only one dispatches"
- "polling run N+1 runs while N is still mid-dispatch; N+1 sees N's claim and skips"
- "the cursor advances correctly even when Linear returns issues in non-deterministic order"
- "a missed cycle (cron skipped at T=10) is recovered by the next cycle (T=15) without losing transitions"

Each of those is a one-paragraph test against the fake, *if* the design has a clock seam and a "drive one polling cycle" entry point. If the design instead wires polling directly to `setInterval` or to GitHub Actions' cron exclusively, those tests become e2e or don't get written. **The DESIGN.md decision worth naming**: there should be a `pollOnce(deps)` function that the GitHub Action calls in production and the test harness calls with a fake clock + fake Linear. That seam is load-bearing for testability.

**3. Idempotency keys are a test contract, not just an implementation detail.** PLAN.md flags "per-issue idempotency keys" as the race mitigation. The DESIGN.md should specify the key derivation as a pure function with documented inputs (issue ID, workflow state, transition timestamp, attempt #?) so it can be tested independently. The risk: idempotency keys derived inline at dispatch time, with implicit dependencies on `Date.now()` or process state, can't be tested for the property that actually matters — *the same logical transition produces the same key on retry*. That's the bug a test should defend against, and it only works if the key derivation is a function with no hidden inputs.

Same shape applies to the cursor model (one of your open questions): if the cursor is "the highest-numbered transition ID we've seen," it's testable; if it's "a blob of state in a Linear custom field updated by a side effect inside the polling loop," it's much harder to test in isolation. **Press toward the testable shape.**

**4. The ephemeral-machine layer can't be unit-tested but needs explicit policy on what gets simulated vs. left for the dogfood phase.** Coder CLI spin-up, auth distribution, teardown reliability, budget enforcement — these are real distributed-system risks, and there's no honest way to test "does the machine actually tear down on time" without spinning up a real machine. DESIGN.md should be explicit about:

- What's tested at the contract level (e.g. "the harness invokes `coder workspaces create` with these args given these inputs" — testable against a mock CLI)
- What's tested against a real Coder dev instance, on what cadence (nightly? pre-release?), with what budget cap (you do NOT want the test suite itself blowing through ephemeral-machine cost — same risk as the production system)
- What's intentionally only verified in dogfood (e.g. "the external reaper catches stuck machines after 4h" — you genuinely cannot test this on a CI timeline)

The thing I'd flag as a hidden test-cost trap: if the test suite spins up real Coder machines, the test suite itself is now a budget-cap concern. DESIGN.md should treat the test infrastructure as one of the systems whose cost the budget cap covers, not as exempt.

**5. The auth-distribution risk needs a test that asserts the negative.** PLAN.md correctly flags auth scope, expiration, and at-rest storage as concerns. The hardest part to test is *the leak that didn't happen*: "the Anthropic API key was not written to a place that survives machine teardown," "the Linear API key scoped to project A was not used to dispatch against project B." These are negative assertions, and they're the bugs you find in production by reading a postmortem.

The design decision worth pricing: do auth credentials flow through a single `getCredentials(taskId, scope)` chokepoint that tests can assert against, or are they read directly from env vars at the point of use? The former is testable ("we asked for Linear-scoped-to-project-A credentials and we got back exactly that, not a broader scope"); the latter requires you to grep the codebase to convince yourself nothing leaked. **DESIGN.md should specify the chokepoint pattern explicitly** so the test surface for "no auth leak" is a single function with a single test suite, not a scattered audit.

**6. Linear API rate-limit behavior needs a test, not just a design rule.** PLAN.md sets the steady-state polling design — narrow queries, modest cadence — but the *failure mode* (what happens when we hit the cap anyway, mid-dispatch, partway through a batch?) is what'll bite. The cheapest test that defends this: fake Linear returns 429 / complexity-cap responses on demand, and the polling loop's behavior under that condition is asserted. Specifically:

- Does the loop back off, or does it spin?
- Does a partial batch leave the cursor in a coherent state, or does it advance past undispatched issues?
- Does a 429 during dispatch (rather than during polling) correctly leave the issue un-claimed so the next cycle retries?

These are testable against the fake Linear with a "configure to return 429 for the next N calls" knob. The risk if untested: the first real rate-limit event in production is the test, and the failure mode is "we lost transitions."

**One meta-note for DESIGN.md structure**: there's a temptation in distributed-system designs to put "testing" as a single section near the end. Resist that — the testability constraints I'm naming above are *load-bearing on the architecture*. The clock seam, the idempotency-key purity, the credential chokepoint, the fake-Linear layer — these are design decisions, not test-suite decisions, and they belong inline in their respective sections (orchestration model, auth, polling architecture) with a one-line "tested via X tier" note. The failure modes section then references the test contracts rather than restating them.

The pattern same shape as "test the boundary, not the implementation": the DESIGN.md should design *for* testable boundaries, and the test plan falls out of that rather than being bolted on.

### From whiteboard-design-systems

I want to push you, DESIGN.md author, on **the vocabulary surface** — because linear-loom's API isn't TypeScript types or React props, it's the *words* that show up in five different surfaces simultaneously: Linear custom-field values, label strings, CLI verbs, slash command names, and the agent-flow adapter contract. Those five surfaces describe the same five concepts. If they drift, every operator and every agent has to maintain a translation table in their head. Let me name the load-bearing decisions I'd want the doc to commit to:

**1. The flow-type vocabulary is the public API. Treat it that way.**
PLAN.md lists `planning`, `iterative-confidence`, `iterative-interactive`, `whiteboarding`, `evaluator-panel`. Five names, three different morphologies (`planning` is a gerund, `iterative-confidence` is an adjective-noun compound naming a *loop variant*, `evaluator-panel` is a noun-noun compound naming a *thing*). That inconsistency is going to bite. The operator picks a label from a dropdown — they need the names to be parallel. Either pick a consistent shape (all gerunds: `planning`, `confidence-iterating`, `interactive-iterating`, `whiteboarding`, `panel-evaluating` — admittedly ugly) or all noun-form flow names (`plan`, `confidence-loop`, `interactive-loop`, `whiteboard`, `evaluator-panel`). I lean toward the noun form: a Linear label answers "what *kind* of work is this issue?" not "what verb is being performed?" Whatever you pick, **commit to one morphology and name the rule in DESIGN.md** so future flow types ( `griot-capture`? `adr-record`? ) don't introduce a sixth shape.

**2. Decide once whether the flow-type is the *flow* or the *underlying skill*, and pick the level of abstraction deliberately.**
Right now `iterative-confidence` is named after the substrate skill (`/ev-loop-confidence`) and `whiteboarding` is named after an activity. That's a semantic-vs-literal mismatch with consequences: if `/ev-loop-confidence` is renamed or replaced by a successor skill, do all Linear issues retroactively become wrong? You want flow types named at the **semantic-intent level** ("the operator wants iterative agent work with confidence-gated unit progression") rather than the **implementation level** ("this triggers the `/ev-loop-confidence` skill"). The mapping flow-type → composed skills lives in the adapter, not in the name. Semantic survives a skill rename; literal doesn't.

**3. CLI verb shape — match an existing vocabulary, don't invent a parallel one.**
PLAN.md mentions `linear-loom project create` and `linear-loom poll` as exemplars. Loom already has its own verb vocabulary (`loom plan`, `loom unit`, `loom pr open`, etc.). If linear-loom invents `linear-loom dispatch`, `linear-loom claim`, `linear-loom reap` while loom uses different verbs for analogous concepts, you've created the operator-confusion risk you flagged in PLAN.md's Risks section as a self-fulfilling prophecy. DESIGN.md should pick: either (a) deliberately mirror loom's verb shapes where the concepts overlap (project bootstrap, status transitions, event recording) and only diverge where the orchestration model demands it, or (b) explicitly declare linear-loom uses an orchestration-flavored vocabulary distinct from loom's pairing-flavored vocabulary, and name the rule. Either is defensible. Drifting into accidental divergence is the failure mode.

**4. The slash-command surface and the CLI verb surface are different audiences. Don't conflate them.**
`/linear-loom-plan` is a slash command an *agent* invokes inside a session. `linear-loom poll` is a CLI verb an *operator or GH Actions runner* invokes from a shell. These names should not be mechanically parallel ("for every CLI verb there's a matching slash command") because the audiences and contexts differ. A `/linear-loom-poll` slash command makes no sense (agents don't poll their own queue). A `linear-loom plan` CLI verb might make sense as a "create a plan-type issue from the terminal" affordance — but only if that operator workflow exists. Let each surface name what it actually does for its actual user, and resist symmetry for symmetry's sake.

**5. Custom field vs label for flow-type encoding (open question in PLAN.md) — this is a semantic-contract decision, not just a UI/polling tradeoff.**
Labels in Linear are loose, free-form, multi-valued, and shared across all issue types in a workspace. Custom fields are typed, single-valued (for enum fields), and structured. If the flow-type is "the contract that determines which adapter runs," it's a **typed enum**, not a tag. A label saying `planning` mixed in with other free-form labels (`urgent`, `frontend`, `q3`) loses the property that "exactly one flow-type per issue, drawn from a closed set" — which is what the dispatcher needs to be correct. Custom field is the right semantic shape; labels are the right visual-affordance shape. If you want both, DESIGN.md should name **one as the source of truth** (custom field) and the other as a derived visual mirror (label auto-synced from custom field), not let them be two parallel writable surfaces that can disagree.

**6. The decision-vs-process axis from RESEARCH.md is the storage-naming contract. Surface the categories explicitly in DESIGN.md.**
RESEARCH.md sorts artifacts by which side of the axis they fall on (decision → git; process → Linear). DESIGN.md should name **the two categories as first-class concepts** with stable labels, and place every artifact type (PLAN.md, adr-log, RESEARCH, whiteboards, retros, INTERVIEW, events, status, check-ins, units) in exactly one bucket. The categories I'd lean toward: `decision-artifacts` (git, diff-able, evolution matters) and `process-artifacts` (Linear, audit-trail, latest-state matters). When someone in six months proposes a new artifact type, the bucket-question forces the right placement automatically. Don't make this implicit. The axis is the architecture; name it.

**7. Reserve a namespace for flow-type extensibility from day one.**
PLAN.md notes the taxonomy is extensible. If flow-types are encoded as a custom-field enum, the value-set is closed and extending it is a Linear admin operation. If they're labels, the value-set is open and any operator can invent a new flow-type that the dispatcher doesn't know about. Either model needs an explicit answer to: **what does the dispatcher do when it sees an unknown flow-type value?** Options: skip silently (dangerous — work doesn't get done), error loudly (annoying — blocks operators experimenting), or dispatch to a `fallback` / `unrecognized` flow that comments on the issue ("I don't know how to do `griot-capture` yet"). I'd push for the third — it's an off-the-rails escape hatch that's *explicit and named* rather than a silent failure. Whatever you pick, name the policy.

**One thing I'm explicitly not pressing on:** the Coder CLI vs alternatives comparison, polling cadence math, auth secret distribution. Those belong to substrate-engineer and security-flavored lenses, not mine. I'm here for the words on the labels and the verbs in the CLI.

### From whiteboard-react-architect

The literal React-API lens doesn't apply — there's no `'use client'` boundary, no component tree, no hook composition. But the *architectural shape* questions my lens trades in — composition vs. configuration, where state lives, how the call site reads, when a thing is doing two jobs and should split — translate cleanly onto CLI verbs, skill adapters, and the cursor model. I'll engage on that translated basis and stay narrow; I'll defer the substrate-specific stuff to `whiteboard-substrate-engineer` and the throughput/cost stuff to `whiteboard-performance`.

The composition-shape lens, ported off React and onto CLI verbs + flow adapters, finds three things in this design that the DESIGN.md author has to settle deliberately rather than drift into.

**One: the adapter contract is the load-bearing API surface, and it deserves the same scrutiny a component prop API would get.** PLAN.md (`Risks`, "Agent flow adapter divergence") already names it, but doesn't price the actual shape question. Five flows (`planning`, `iterative-confidence`, `iterative-interactive`, `whiteboarding`, `evaluator-panel`) plus an extensibility promise — that's exactly the moment my React lens screams "composition over configuration." The wrong shape here is one fat `FlowAdapter` interface with optional fields for every flow's quirks (`whiteboardRounds?: number`, `confidenceThreshold?: number`, `panelLenses?: string[]`). That's the prop-heavy monolith antipattern: every new flow widens the contract, every old flow has to read past fields it doesn't use, and the sixth flow type breaks the abstraction. The right shape is a *minimal* shared contract — `dispatch(issue) → run → report(status, artifacts)` — and per-flow modules that compose substrate skills however they need. Symmetrical at the dispatch boundary, asymmetric in the body. DESIGN.md should write the contract explicitly and resist the urge to put flow-specific knobs in it.

**Two: the cursor / "what have we already dispatched" question is a state-location question, and the ladder matters.** PLAN.md lists three candidates as a flat menu (Linear custom field, repo state file, Coder-side ledger). My lens reads that as the classic local-vs-lifted-vs-URL-vs-server choice, and the analogous ladder applies. Linear custom field is the "URL state" tier — authoritative, shareable, survives all client restarts, but constrains the shape (you're writing to someone else's schema). Repo state file is the "lifted local" tier — yours to shape, but now you've got a write-back loop from a polling GitHub Action that has to commit to the repo, which is its own pile of failure modes. Coder-side ledger is "ephemeral local" — fastest, but lost on Coder outage and useless for cross-run idempotency. **My lean: Linear custom field, because Linear is already the source of truth for issue status and the cursor is just a view of that truth**; "have we dispatched this transition" is a per-issue fact, store it next to the issue. Repo state file pulls a second source of truth into a system that's actively trying to make Linear *the* source of truth — that's the inconsistency this design has to avoid. DESIGN.md should make this argument explicitly, not just pick.

**Three: the CLI verb surface reads better as a family of small verbs than one polymorphic `linear-loom` god-command.** PLAN.md gestures at `linear-loom project create`, `linear-loom poll`, etc., but the design moment is whether `poll` is one verb that does discovery + claim + dispatch + cursor-update, or four composable verbs (`discover`, `claim`, `dispatch`, `cursor advance`) that the GitHub Action wires together. I'd push hard for the latter. The composed shape gives you (a) testable units — you can dry-run `discover` against a live Linear workspace without touching anything, (b) an obvious recovery story when something fails mid-cycle (re-run from the failed verb, not the whole pipeline), and (c) a clean seam for the Phase N webhook upgrade — a webhook handler invokes `claim` + `dispatch` directly, bypassing `discover`. One concept per verb. When you can't describe `linear-loom poll` without "and," split it.

**One more, briefer:** the `iterative-confidence` vs `iterative-interactive` split worries me on the naming-as-architecture axis (which is `whiteboard-design-systems`'s lane, so I'll defer to them on the actual rename), but my architecture concern adjacent to it: are these two truly different flows, or one flow with an `interactive: bool` mode? If they share the underlying `/ev-loop-*` skill family and only differ in whether a human is in the loop, that's two adapter modules with 90% shared body — which usually means the *adapter* layer is wrong, and the real shape is one `iterative` adapter that asks Linear "is this issue assigned to a human collaborator or not" and routes accordingly. DESIGN.md should make this either/or explicit.

What I'd want resolved next round: the adapter contract written out concretely (even pseudocode is fine — just enough to see whether it's minimal-and-composable or wide-and-configurable); the cursor decision committed with reasoning; and the verb decomposition either confirmed-composed or argued-monolithic. I'll defer cost-monitoring shape and Coder-vs-Modal tradeoffs to `whiteboard-performance`; I'll defer "should it be webhooks anyway" pressure-testing to `whiteboard-skeptic`; substrate-engineer leads on plugin layout and the linear-loom ↔ loom boundary.

### From whiteboard-a11y

Most of my usual surface area — semantic HTML, focus management, screen-reader experience, keyboard parity, motion — doesn't apply here. Linear owns the operator-facing UI; the polling worker and Coder VM are headless. I'm partial-recusing on the orchestration core. But there are four inclusion-shaped seams in this design that the DESIGN.md author should pin down rather than letting them emerge, because they're cheap to design in now and annoying to retrofit.

**1. Linear comment microcopy is a UX surface, and it's the one we DO own.** The agent's voice in Linear — the dispatch comment, the status updates, the failure comments, the "I'm done, PR up for review" close-out — is read by the operator, by other engineers on the issue, eventually by non-engineering stakeholders (the dossier names PMs, designers as a deliberate audience). That audience is the one place where the "plain language wins" principle has real teeth in this project. The DESIGN.md should commit to a comment shape per flow with concrete templates, not "the agent will comment on the issue when relevant." Specific decisions to make explicit: does a failure comment lead with the failure or with the recovery action? Does a status transition comment repeat what Linear's own activity feed already shows, or stay quiet and let Linear's native UI carry it? When an agent posts a 400-line stack trace, is that in the comment body or a collapsed details block or an attached file? Each of those choices reads very differently to a screen-reader user paging through the issue, and very differently to a PM scanning for "is this on track." Pick a posture in the design doc.

**2. Operator notification model needs a deliberate choice, not Linear's default.** Linear's `@`-mention and assignee-change behavior fires native notifications (email, Slack, in-app) by default. When this system dispatches 12 ephemeral machines in a 5-minute window and each one comments on its issue, that's 12 notifications to the operator. Some operators have ADHD-friendly notification setups where 12 dings is genuinely disruptive; some are notification-deaf and need the dings to stay aware. The DESIGN.md should name this explicitly: does the agent `@`-mention the operator on every state transition, only on completion, only on failure, or never (operator pulls from the issue view)? "Never" is probably the right default for a swarm — the operator chose Linear-as-orchestration-substrate partly *because* they want to consume on their own cadence — but it should be a design decision in the doc, not whatever the Linear API does by default when an agent comments.

**3. Failure surfacing in Linear: distinguishable without color.** The Symphony-style state machine (`Todo → In Progress → In Review → Done`) is fine, but it doesn't model failure states cleanly. A machine spin-up failure, a Coder CLI outage, an agent flow that hit max-iterations without converging — those need to be visible *as failures* in whatever Linear view the operator is using, not buried in a comment thread on an issue still labeled "In Progress." Two design decisions: (a) is there a dedicated `Failed` workflow state, or do failures route back to `Todo` with a label, or do they sit as `In Progress` with a failure comment? (b) whatever signal we pick — workflow state, label, custom field — it can't rely on Linear's color alone, because Linear lets users theme their workspace and color-blind operators may see two states as visually identical. Use the state *name* and any label *text* as the load-bearing signal, color as redundant decoration only. This is the one classical-a11y concern that survives the recusal.

**4. Operator override / kill-switch ergonomics.** Failure mode #8 in the brief is "agent flow failure." There's an implicit eighth-and-a-half: "operator wants to stop a running flow." If a Coder machine is mid-execution and the operator realizes the issue was mis-labeled, or the agent is clearly off the rails, what's the verb? Linear-side (close the issue, change the state, remove a label)? CLI-side (`linear-loom kill <issue-id>`)? Both? The DESIGN.md should name a single canonical kill path and commit the polling worker to honoring it — otherwise the operator's "stop" intent is at the mercy of whatever the agent happens to check next. Inclusion-wise, this matters because the operator who most needs a fast, low-friction stop is the one who's overwhelmed or panicking — and that's exactly the moment when "remember to also run `coder workspaces stop`" fails them. One verb, one place, documented in the design.

**Defer to siblings** on: idempotency / cursor design (substrate-engineer's territory), Coder CLI auth distribution shape (substrate + skeptic), Linear API rate-limit pricing arithmetic (performance, if engaged; RESEARCH.md already prices the 3M/hour ceiling). My picks above are the load-bearing slices where "design for inclusion from the start" actually has purchase in a CLI/orchestration project; the rest is genuinely outside my lane.
