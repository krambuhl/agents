# Whiteboard — shift 02: research as a living fact-canon

**Topic.** Today the substrate treats `RESEARCH.md` as a one-shot input to `PLAN.md`. The operator wants it to become a canonical fact-store: `/loom-plan` aggressively delegates fact-gathering to `/loom-research` in fresh-context sub-agents (keeping planning context clean); `PLAN.md` references `RESEARCH.md` by anchor; implementation phases via `/ev-loop-interactive` can AMEND `RESEARCH.md` when new facts surface mid-flight.

**Substrate evidence.**
- `plugins/loom/skills/loom-plan/SKILL.md:89-94` — auto-spawns `/loom-research` only on `RESEARCH.md` absence; no path for "exists but plan-interview surfaced a fact gap."
- `plugins/ev/skills/ev-loop-interactive/SKILL.md:469-485` — inner-RPI hop already spawns `/loom-research` mid-execution; flags an OPEN question on append-vs-overwrite when research already exists.

- `plugins/loom/skills/loom-revise-plan/SKILL.md` — flavor-routing grill pattern (mechanical | research) that a future `loom revise-research` could mirror; listed under Deferred at `loom-research/SKILL.md:402`.

## whiteboard-substrate-engineer

The append-vs-overwrite question is the load-bearing one, and the proposed APPEND-WITH-PROVENANCE answer is the right shape — but only if we make the file genuinely append-only at the write layer rather than by convention. Today `bin/loom research` is a `copyFileSync` over `RESEARCH.md` gated by a committed-state check; that's overwrite-by-default with a "don't do that" guard. An `--amend` flag (or sibling `bin/loom revise-research` verb) needs to invert this: read existing, append a delimited block with structured provenance frontmatter (project slug + phase + amending session id + timestamp), write the concatenation, commit. The provenance metadata is the substrate's append-only invariant materialized in markdown — without it, two amendments to the same section become indistinguishable and the "fact canon" is just a longer freeform doc. Critically: amendments append blocks, never edit prior blocks. If a prior fact is wrong, the amendment is a correction-block that supersedes; the prior block stays for forensics. Same instinct as the event log — current state is a fold over blocks, not a mutation in place.

Concurrent-write safety has two regimes worth separating. **Same-PR amendments** (one `/ev-loop-interactive` session amending RESEARCH.md mid-phase) inherit the existing single-writer assumption — one ev-loop owns the working tree, one amend at a time, the read-append-write race is local and bounded. **Cross-PR amendments** (two parallel ev-loop sessions on the same project, each on its own branch, both amending) is where the substrate gets genuinely tested. The append-only block shape makes this *git-mergeable* by construction: two append-blocks at the end of RESEARCH.md merge cleanly the way two appends to a JSONL file merge cleanly. That's the design win you cash in by going append-only — concurrency is solved by git, not by the verb. If amendments could edit prior blocks, every cross-PR amendment becomes a merge conflict and the operator pays the cost. Worth noting: this is the same shape `manifest.toml` `[[events]]` blocks use today, and they merge fine in practice for the same reason. Family-shape consistency for free.

On schema evolution: yes, the moment PLAN.md cites RESEARCH.md by anchor, RESEARCH.md needs a soft schema — heading conventions at minimum, ideally a `## Facts`, `## Open questions`, `## Decisions` taxonomy with stable anchor slugs. Without that, anchor citations rot the first time someone reorders a heading. Lean soft-extension: a recommended skeleton emitted by `bin/loom research` when it scaffolds, but the loader stays tolerant of freeform sections (read what's there, don't validate structure). The amend-block frontmatter is the one place to be strict — `<!-- amendment: slug=X phase=Y at=Z -->` or equivalent — because that's what programmatic readers (a future "show me all amendments by phase" query) will key on. Keep the prose layer freeform; make the provenance layer structured. That split is also what lets the schema evolve without a hard migration: existing committed RESEARCH.md files have zero amendment blocks and read identically under the new schema.

For family-shape: `loom revise-research` as a sibling of `loom revise-plan` is the right naming, and it should inherit the flavor-routing grill pattern but with a *different* flavor axis. `revise-plan` splits on `mechanical | research`; `revise-research` would split on something like `correction | new-finding | clarification` — the kinds of amendment that actually surface in implementation. The CLI verb stays a CRUD primitive (append-block-with-provenance-and-commit, nothing more); the flavor-routing grill lives in the skill body, same shape as `loom-revise-plan` already does. The substrate gap to name explicitly: today there's no `revision-trigger` classification on session-note captures (the ev-loop SKILL.md flags this at line 466 as a Phase 7 follow-up). If amendments become a real workflow, that classification stops being optional — every amendment should leave a session-note breadcrumb with the trigger, so the retro can fold "what facts did we learn mid-flight" without re-reading the diff. The amendment block in RESEARCH.md and the session-note capture are the same event told twice, which is fine, but the substrate should make it cheap to emit both.

On cost-of-substrate, I'd push back gently on the "PLAN.md embeds corrections + retro captures learnings" alternative. That keeps the substrate weight low but pays the cost in *fact discoverability across projects*. The current shape already has the gap: when a future project's `/loom-research` is investigating topic X, it can't discover that a prior project amended its RESEARCH.md mid-flight with a correction about X — because the correction lives in PLAN.md prose or a retro, neither of which research agents read systematically. The bidirectional flow's real payoff isn't in-project (the operator can already read the PR), it's *cross-project fact reuse*: RESEARCH.md becomes the durable, queryable substrate that the next project's research phase grounds against. That payoff is only worth the substrate weight if amendments are first-class queryable artifacts (provenance frontmatter, append-only, stable anchors) — half-measures get you the weight without the payoff. So: do it, but commit to the structured-provenance shape, otherwise the simpler "embed in PLAN.md" answer is correct.

## whiteboard-design-systems

**The section headings are about to become an API, and the current naming leaks the interview process into the public surface.** Once RESEARCH.md is referenced by anchor from PLAN.md and AMENDED mid-execution by `/ev-loop-interactive`, every `## Whiteboard contributions` and `## Shift N — <topic>` heading becomes a citation target — `RESEARCH.md#whiteboard-contributions` is a stable link other artifacts will hold. The problem: those heading names describe *how the facts got gathered* (a whiteboard panel ran; a domain shift fired), not *what facts they are* (token-pipeline constraints; on-call-rotation impact; Postgres-write-amplification numbers). That's the same antipattern as `<BlueButton>` vs `<PrimaryButton>` — naming after the visual/process artifact rather than the semantic role. A referenced fact-canon wants headings shaped like `## Constraints — token pipeline`, `## Observed costs — CI runner minutes`, `## Open hypotheses — sharding strategy`. The interview-shape provenance still belongs *somewhere* — that's exactly what `RESEARCH-NOTES.md` is for. Keep the literal-process structure in NOTES; promote the semantic-fact structure to RESEARCH. The split already exists; the heading vocabulary just hasn't followed it across yet.

**On composition vs configuration: do not build `loom research --append --from-phase=N --citing-evidence=...`.** That's the s-expression-flattened-into-a-prop-matrix shape — every new flow grows another flag, and the verb's contract becomes "whatever the union of all callers needed." The composable shape mirrors what the substrate already does well: small primitives that callers assemble. The minimum primitive set I'd lean toward: `loom research-append <slug> --section=<semantic-heading> --fact-file=<path> --citing=<phase|adr|evidence-url>` (the low-abstraction escape hatch — adds one grounded fact under a named section, no interview) and a thin `loom research-amend <slug>` that wraps `/loom-research --mode=amend` for the high-abstraction "fresh interview, but starting from the existing canon" flow. The bidirectional `/ev-loop-interactive` hop becomes a *caller* of `research-append` when the trigger is a single grounded fact, and a caller of `research-amend` when the trigger is a whole new domain. The rule that earns its keep: **bypassing the interview is allowed, but every append still carries a citation** — the off-the-rails escape hatch stays inside the system because the contract (every claim cites a source) is enforced at the verb layer, not the skill layer.

**Vocabulary cohesion: `loom revise-research` is the wrong shape against the existing verb family.** Today's verbs split into two patterns — single-word lifecycle verbs (`plan`, `research`, `checkin`, `doctor`, `pr`, `phase`, `adr`) and noun-then-subverb verbs (`retro write`, `retro read`, `retro list`, `project read`, `events append`). `revise-research` is a third shape (hyphenated compound verb) and it reads like an action grafted onto a noun with a dash where the rest of the CLI uses a space. The cohesive options, in order of preference: `loom research amend <slug>` (noun-then-subverb, matches `retro write`; `amend` is the precise verb for "add to a committed thing without rewriting it" and lines up with git's vocabulary the operator already holds); `loom research revise <slug>` if "revise" is the deliberate semantic (note: amend ≠ revise — amend adds, revise rewrites; pick the verb that matches what the verb actually does); avoid `loom research --revise` because flag-as-mode-switch hides the second verb from `--help` discovery and reads as a modifier on the primary verb rather than a sibling operation. Once `research` becomes a noun-with-subverbs, the natural shape opens up: `loom research init` (today's `loom research`), `loom research amend`, `loom research append`, `loom research show`. That's the same shape as `retro` and `project` — cohesion compounds.

**On the high-low abstraction parallelism: yes, ship both, and name the seam.** The high abstraction is `/loom-research` itself — fresh-context sub-agent, grill-me interview, whiteboard-per-shift, fact-check gate. The low abstraction is the `loom research append` verb above — single fact, single citation, no interview, no panel. The two should share *one* invariant: every fact in RESEARCH.md has a source. The interview enforces it conversationally; the append verb enforces it via a required `--citing` flag that refuses empty. The trap to avoid: a third abstraction in the middle — a "lightweight interview" mode that's neither a full grill nor a single append. That's where prop-matrix bloat starts. Two clear tiers with a sharp seam beats three blurry ones. And one note for the eventual implementer: if amend-mode RESEARCH.md grows new semantic sections that the current section vocabulary doesn't cover (e.g., `## Mid-execution discoveries`), that section-vocabulary extension is a real authoring step worth naming in the unit contract, not something to discover at write time.
