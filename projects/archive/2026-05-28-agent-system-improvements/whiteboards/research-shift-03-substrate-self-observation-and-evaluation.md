# Whiteboard — shift 03: substrate self-observation and self-evaluation

**Topic.** The substrate is data-free about its own use and has no regression-testing framework for its own components (except learnings, via griot). Two related fix shapes surfaced: (1) cross-project event aggregation (`loom stats --since=7d` or similar); (2) extend the griot-shaped benchmark loop to evaluators, whiteboard engineers, and skill outputs.

**Substrate evidence.**
- `plugins/loom/cli/verbs/loom/events.ts:13-27` — `readEvents` reads per-project events filtered by `--since`, `--event`, `--limit`. No cross-project aggregation.
- `plugins/commons/cli/lib/events.ts` — canonical event-emission helper, synced into every plugin's `cli/lib/`.
- griot agents (`griot-judge`, `griot-rewriter`, `griot-operator`, `griot-rubric-author`, `griot-debate-summarizer`) — full evaluation loop for learnings. No equivalent for evaluators or whiteboard engineers.
- `scripts/sync-shared.ts --check` — existing file-mirror invariant enforcement, the closest existing analog to convention-drift detection.

## whiteboard-substrate-engineer

**Append-only invariant under aggregation.** Good news first: nothing in option 1 threatens the append-only invariant, because aggregation is a pure read-fold over `manifest.events` (the TOML store at `plugins/loom/cli/verbs/loom/events.ts:12-26`) plus the legacy `events.jsonl` reader in `plugins/commons/cli/lib/events.ts:11-52`. The aggregator's job is to fold N per-project event streams into one cross-project view; it writes nothing back into project state. The real question is **where the cache lives if we don't want to re-fold every project on every invocation.** Three honest options, in increasing cost-of-substrate: (a) no cache, accept O(projects × events) on every call — fine until we have hundreds of projects; (b) a per-invocation in-memory fold with no persistence — same answer; (c) a sibling artifact under `~/.loom/` or `projects/.cache/` keyed by project-slug + last-event-timestamp, invalidated when a project's tail moves. Option (c) is a new artifact family and deserves a deliberate decision before it appears — once a cache exists, every future writer has to remember to invalidate it, which is exactly the kind of "parallel-safe by convention, not by construction" trap I'd want to avoid. My lean: ship (a) until cardinality forces (c), and when (c) lands, make it a derived artifact (regenerable from a `loom stats --rebuild`), not a source of truth. The manifest stays canonical; the cache is a hint.

**Family-shape consistency.** Looking at the existing verb roster in `plugins/loom/cli/verbs/loom/` — `events`, `checkin`, `plan`, `phase`, `pr`, `session`, `retro`, `adr`, `doctor`, `research`, `project` — the family pattern is **noun-verb, where the noun names a substrate artifact and the verb names the operation on it**. `events` already owns the events-shaped read surface (`read`, `latest`); cross-project aggregation is still events-shaped. So the consistent shape is `loom events aggregate --since=7d --cross-project` or `loom events stats`, not a sibling `loom stats` noun. A new `stats` noun signals that observability becomes its own artifact family — and if we go there, the substrate should know what artifacts live in that family before we name the first verb. Otherwise `loom stats` becomes a junk drawer: today it's event rollups, next quarter it's checkin counts, then panel-spawn frequencies, then evaluator hit-rates, and now `stats` means "anything observability-flavored," which is the design-systems "literal-vs-semantic naming" smell applied to verbs. Lean: `loom events` grows aggregate sub-verbs; `loom stats` waits until there's a real cross-noun aggregation surface (and then it's likely `loom report` or `loom observe`, not `stats`).

**The griot extension.** The griot loop maps cleanly onto evaluators in principle but the substrate analog is asymmetric in a way worth naming. For learnings: the artifact under test (a learning markdown file) is *the thing itself*, and the rubric scores its quality. For evaluators: the artifact under test is the evaluator's *behavior* — a function from input artifact to flag-set — and the rubric is a corpus of `(input, expected_flags)` pairs. That's not a rubric, that's a **fixture-driven regression suite**, which is a fundamentally different shape: rubrics are open-ended judgment, fixtures are pass/fail. The griot loop's middle pieces still translate (rewriter amends the evaluator body, operator diagnoses why a regression appeared), but `judge` becomes something more like `runner` because the scoring is mechanical. So I'd resist the "just generalize griot" framing. A `benchmark-*` family is genuinely a separate substrate family with its own corpus artifacts (probably under `plugins/<plugin>/benchmarks/<evaluator-name>/cases/*.json` with mirrored expected-output files), its own runner verb (`guild benchmark run evaluator-tokens`), and only shares the rewriter/operator agents *as composition*, not as inheritance. Same loop-shape, different substrate. Naming it `benchmark-*` keeps griot meaning "learnings-quality loop" and leaves room for both families to evolve independently. If we collapse them into one griot, we'll spend the next year explaining why `griot-judge` sometimes scores prose and sometimes runs fixtures.

**Cost-of-substrate and sequencing.** Option 1 (lightweight metrics) costs almost nothing — a new sub-verb that read-folds existing append-only state, no new artifact family, no cache, no migration. Option 2 (self-evaluation suites) is a whole new substrate family: corpus artifacts, expected-output files, a runner verb, a rewriter loop, a regression-diagnosis surface, plus the maintenance cost of keeping the corpus current as evaluator contracts evolve. Option 2 doesn't pay rent until we have (a) enough evaluators that one breaking another is a real risk, (b) enough contributors that no single person remembers all the contracts, or (c) enough churn in evaluator bodies that regressions actually happen. We're not there yet — the evaluator roster is small enough that the contract-fit lens plus targeted unit tests cover the failure modes. **Ship option 1 first, and let it tell us whether option 2 is needed.** Specifically, the metrics from option 1 are the input signal for "is the evaluator corpus drifting" — if `loom events aggregate` shows evaluator-X-blocked-but-overridden climbing month over month, *that's* the substrate evidence that says "now we need the regression suite." Building option 2 before option 1 is building the cure before measuring the disease.

**Convention drift as the cheapest baseline.** This is the move I'd actually push hardest on, because it's the smallest substrate change with the highest leverage. `scripts/sync-shared.ts --check` already enforces the file-mirror invariant — that every plugin's `cli/lib/events.ts` is byte-identical to `plugins/commons/cli/lib/events.ts`. The natural extension is a behavioral-convention check that doesn't need new substrate at all: a `scripts/check-conventions.ts` (or a `commons doctor --conventions` verb if we want it CLI-shaped) that asserts things like "every plugin that emits events imports from `cli/lib/events.ts`, not a local fork," "every `evaluator-*` agent has the expected frontmatter keys," "every `whiteboard-*` agent declares its boundary section." This is the substrate analog of a lint rule: idempotent by construction, additive-only (each new convention is one more assertion), and the cost lives at check-time, not at session-time. It surfaces drift before drift becomes regression, which is exactly the observability question option 1 and option 2 are circling. I'd land convention-drift first (one afternoon), `loom events aggregate` second (one week), and benchmark-* substrate only when the drift checks plus aggregated metrics tell us a specific evaluator is regressing in the wild.

## whiteboard-testing-strategy

The substrate has three tiers of failure for an evaluator, and they want three different test shapes. **Tier one: rubric-body coherence** — does the prose rubric in the evaluator's frontmatter actually match the checks the body performs? This is unit-shaped, pure-function, cheap. Defends against the most common silent regression: someone edits the body to add a new check but forgets the rubric, or tightens the rubric without updating the body. **Tier two: flag accuracy on a corpus** — given a known input, does the evaluator emit the expected findings at the expected severity? This is integration-shaped and expensive; defends against logic regression in the check itself. **Tier three: panel verdict stability** — for a fixed input project, does `guild-validate` produce the same blocking/advisory verdict it did last week? End-to-end, slowest, most fragile; defends against emergent regressions where individual evaluators are fine but composition shifts. The risk-to-tier mapping matters because tier three is what people want to build first and is almost always the wrong starting point — it produces flaky signals that get muted, and the noise hides tier-one drift the cheap test would have caught immediately.

On corpus shape: pure fixtures rot fast here because the evaluator surface is unstable by design. Today's `evaluator-tokens` knows about CSS Modules; in six months it knows about Panda or whatever. A `(diff, expected-flags)` fixture file gets stale the moment the rubric gains a new check, and the test starts failing not because the evaluator regressed but because the fixture is from the old world. Pure factories drift the other way — synthetic diffs miss the weird-shaped real bugs that motivated the evaluator in the first place. The hybrid that fits: **factories for the negative space** (synthetic clean inputs that should produce zero findings — these test recusal and false-positive rate, and they're stable because "clean" is stable), **fixtures for the positive space** (a small curated corpus of real-world diffs that historically tripped the evaluator, each tagged with the finding it should still produce). The fixtures rot, but they rot loudly and visibly — when one stops matching, you decide whether the evaluator regressed or the world moved on, and you update or retire the fixture. That decision is the signal, not noise.

On what to skip: a flag-by-flag accuracy suite for every evaluator is the trap. The load-bearing question isn't "did this evaluator catch every bug in the corpus" — that's a coverage metric in disguise and it'll get gamed. The load-bearing question is **"is this evaluator still doing work at all?"** Two cheap metrics from `events.jsonl` would catch most real regressions: **spawn-to-finding rate** (over the last N invocations, what fraction produced any findings — sudden drop to zero means the evaluator silently broke or its prompt collapsed) and **non-applicability rate** (what fraction recused — sudden spike to 100% means the evaluator stopped recognizing its own domain). Neither requires a corpus; both fall out of aggregating events you already have. The full benchmark loop is the right second move, not the right first move.

On parallel safety: `events.jsonl` being append-only is the easy half — concurrent ev-loops writing to per-project event logs don't race because they don't share files. The hard half is the aggregator. If `loom stats --since=7d` walks `projects/*/events.jsonl` and a project mid-flight is appending while we read, we get a partial-line read at the tail or a truncated final event. The fix is the same one tail-following tools use: read whole lines, drop any trailing partial, and treat the file as a stream of completed records. No aggregator cache to start with — caching is a premature optimization here, and a stale cache during active ev-loop runs is exactly the consistency bug that erodes trust in the stats output. If `loom stats` becomes slow enough to need a cache, the cache key should be `(project, last-event-timestamp)` so concurrent writes invalidate cleanly. The deeper parallel-safety question is whether `events.jsonl` records carry enough provenance — session id, evaluator name, contract id — to reconstruct a coherent timeline when interleaved across projects; if not, the aggregator can't produce meaningful cross-project rollups regardless of caching.

On cheapest reliable signal: before any benchmark loop, generalize the `sync-shared.ts --check` pattern. That script defends a coherence invariant (commons source == consumer copy) with a tiny script and a CI gate. The analogous invariant for evaluators is **rubric-body coherence**: every check named in the rubric frontmatter is reachable from the body, every check the body performs is described in the rubric. A static analyzer over `agents/evaluator-*.md` that parses the rubric section and greps the body for corresponding language gets you 80% of regression defense for a fraction of the cost of a corpus suite. Same shape for whiteboard engineers: every "what you lean toward" bullet should have a corresponding "what you don't do" boundary, every sibling-engineer reference should resolve to an actual sibling file. These are mechanical checks producing sharp pass/fail signals, runnable in CI in milliseconds, and they catch the most common drift class — silent edits that break the contract a downstream skill relies on. Build the convention-drift detector first; build the benchmark loop only after the cheap signal stops being enough.
