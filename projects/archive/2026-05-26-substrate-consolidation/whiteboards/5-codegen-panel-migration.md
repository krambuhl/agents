# Whiteboard: Codegen + panel migration: guild generate design, deletion sequencing, tests

## Round 1

### From guild:whiteboard-substrate-engineer

Phase 4 did the hard part well: the source fragments are already reference-free (the `personality-base.md` I read inlines as an opening section — its old dispatch-time "read the domain file" mechanism is gone, replaced by "all three sections are present below in this agent"), the tools-map encodes the no-permissive-default contract in its own header, and the manifest enumerates the needed combinations rather than the 240 cross-product. That means Phase 5's `guild generate` is a *fold*, not an orchestration — and that framing is the spine of everything below.

**1. `guild generate` is a CRUD-shaped fold — register it like the other verbs, resist orchestration creep.**

The verb belongs at `plugins/guild/cli/verbs/guild/generate.ts`, registered in both `VERBS` (the help/dispatch table in `guild.ts`) and `GUILD_VERBS` (the handler map in `index.ts`), following the identical `(rest, ctx) => DispatchResult` shape every sibling verb uses. No new patterns. The reads are pure inputs: `panel.manifest.toml` (which combinations + the `[recipes]` + the `[retained]` list), `tools-map.toml` (phase base ∪ domain grants), and the fragment files. The output is one file per combination. That is read-N, write-M with zero external state — the cleanest possible CRUD primitive, and it should *stay* that clean.

The thing I'd press the implementing unit on is keeping the fold pure. There's a tempting orchestration leak: having `guild generate` also re-derive the panel, run a smoke test, or `git add` its output. Don't. The verb reads source and writes agent files — full stop. Panel derivation is `derive-panel`'s job, staging is the loop's job, the smoke test is a separate test target. The moment `generate` knows about "and then validate the panel" it's stopped being a fold and started being a workflow.

On resolution, the tools-map already dictates the algorithm: `agent.tools = phase.base ∪ {Bash grants from domain.<d>.grants}`, deduped, order normalized by codegen (pick a fixed canonical order and make it deterministic). Two fail-loud assertions the verb must make as its first step, both already written into the source as contracts: every phase a combination names must have a `[phase.<phase>]` row or **error** (never a permissive empty fallback — the silent "key missing → no Bash grant" produces a hollow agent that flags nothing); absence of a `[domain.<d>]` row means *phase-base-only*, a **default, not an error** (css-architecture and naming hit this path by design and must come out as `Read, Glob, Grep` exactly).

Synthesized frontmatter per combination: `name`, `role`, `description`, the resolved `tools:` line, `model: inherit`, and `maxTurns: 5` **only for reviewer-phase agents** (a phase-determined fact — codegen reads `maxTurns` off the phase, not a per-combination flag). The body is the inlined concatenation in the Phase-4 order: base → phase → domain → personality. Because the fragments are reference-free, this is `cat` with a synthesized frontmatter header — no strip-then-inline scalpel. If a future fragment reintroduces a "read this other file" line, the fold breaks and you're back to a parser.

**2. Commit the generated output, into `agents/generated/`.**

Gitignore-and-generate-on-install is the wrong shape. The agents *are the product* — the runtime artifacts the marketplace plugin ships and Claude Code spawns. The marketplace copies files; it doesn't run a build step in the consumer. Gitignoring would mean the installed plugin has an empty `agents/` until someone manually runs `guild generate` — breaking the installed-vs-source model. The generated files have to be present in the committed tree to ride the marketplace-install path. So: **commit them.**

Where: a dedicated `plugins/guild/agents/generated/` subdirectory, not overwriting `plugins/guild/agents/` in place. (a) The hand-authored exceptions (`evaluator-contract-fit.md`, the base files, the source fragments) must not live in a directory codegen owns — a `generated/` subdir is a structural declaration "everything here is codegen output, regenerate don't hand-edit." (b) Idempotency becomes verifiable: run generate twice → byte-identical output (no timestamps in the frontmatter — they make every regen a spurious diff); with output isolated to `generated/`, the regression test is `git diff --stat plugins/guild/agents/generated/` is empty. (c) A `<!-- GENERATED by guild generate — do not edit -->` header in each emitted file closes the loop for the next human/agent.

One naming question codegen must settle deterministically: **what do generated files get named?** Keep the family vocabulary — reviewer-phase combinations emit `evaluator-<domain>.md`, planner-phase emit `whiteboard-<domain>.md`, so generated names are *identical to the baked names they replace*. That means every existing reference (`derive-panel`'s fallback rules, `PANEL-COMPOSITION.md`, the `ev-loop` glob) keeps resolving without a rename wave. If generate emitted `skeptic-a11y-reviewer.md`, you'd break every consumer that names agents by the `evaluator-*`/`whiteboard-*` convention. Keep the phase→prefix mapping; it makes Phase 5 a drop-in replacement, not a rename migration.

**3. The self-hosting deletion hazard — generate-and-commit first, delete baked source LAST (Phase 7, not Phase 5).**

Two distinct copies are in play: the committed source-repo baked files, and the installed marketplace copies that Claude Code actually spawns. The installed copies are frozen at last-install; deleting source-repo copies doesn't change what spawns until re-install. So deleting baked source in Phase 5 doesn't immediately break the running panel — but it deletes the higher-fidelity harvest source, AND the baked guild agents are *also* Phase 7's deletion target. The harvest-first/delete-last principle should extend to them.

Recommended order as discrete units: **generate-and-commit first** (both baked + generated exist; nothing breaks). **Cutover is a re-install + a panel smoke test**, not a deletion — the generated agents become runtime-active only on re-install, and the smoke must run against the *installed* generated agents (testing source-repo files green while the installed copy is stale is the fixtures-mask-real-breaks trap). **Deleting the baked source is the LAST unit, deferred into Phase 7's salt-the-earth pass** — or at minimum gated on a green panel-verdict smoke + the drift-detection test (every baked flag code present in the generated agent). The baked files are the rollback path. Net: Phase 5 *generates and cuts over*; Phase 5 does **not** delete baked source.

**4. whiteboard-skeptic: emit one personality-led planner agent, no domain inlined.** Generating `skeptic × {each design domain} × planner` would emit a dozen near-identical agents differing only in a domain catalog the skeptic doesn't need — cross-product explosion in disguise. Instead emit **one** `whiteboard-skeptic.md` from `skeptic × planner` with no domain fragment inlined; tools resolve to `phase.planner.base` alone. The brief's domain arrives at *dispatch*, not generate time. Codegen tolerates a domain-agnostic combination as a *named exception in the manifest* (not a silent `domains=[]`). On idempotency broadly: the verb is a pure function of source inputs — deterministic combination iteration (sort phase, personality, domain), deterministic tool ordering, no nondeterministic content. Run-twice-same-output is what makes the "git diff empty after regen" test meaningful.

**5. Proposed Phase 5 unit breakdown — generate-and-commit early, delete-never (defer to Phase 7).**

- **U1 — `guild generate` verb (the fold), no output committed yet.** The verb + resolution algorithm + two fail-loud assertions + deterministic ordering + frontmatter synthesis. Unit tests: TOML parse + fold correctness, AND a real-artifact test that resolved tools for `a11y@reviewer` exactly reproduce the baked `evaluator-a11y.md` `tools:` line. The carefully-reviewed setup unit — the verb is the whole risk surface, ships alone.
- **U2 — Generate and commit the core panel into `agents/generated/`.** Run the verb, commit output, add `.gitattributes` marker + per-file generated-header. Idempotency regression test (regenerate → git diff empty); assert every generated reviewer carries `maxTurns: 5` + correct `tools:`. Both generated + baked exist; runtime unchanged until re-install.
- **U3 — whiteboard-skeptic + design-systems recipe + project-local domain escape hatch.** The domain-agnostic skeptic emission, the recipe expansion, and the off-rails project-local path. The new-concept emissions versus U2's straight combinations.
- **U4 — Cutover + panel smoke + record the commit-vs-gitignore decision.** Re-install, run the panel verdict smoke against the *installed* generated agents on a sample diff, write the committed-output decision into the project docs. The drift-detection test (every baked flag code present in generated copy) lands here as the green-light gate for deletion. No deletion in this unit.
- **Deletion of baked `evaluator-*`/`whiteboard-*`/`generator-*` → deferred to Phase 7**, gated on U4's drift test + installed-panel smoke green. If PLAN insists Phase 5 own the deletion, it's a fifth unit landing strictly last, only after U4 is green — but my strong lean is Phase 7, so Phase 5 stays "codegen exists and is the runtime" as one arc.

I'd resist two collapses: don't merge U1 and U2 (verb-correctness vs output-commit are different conceptual changes), and don't let U2 quietly delete baked files to "make room" (the deletion hazard sneaking into a generation PR). Generate-first, cutover-second, delete-last — the harvest-first/delete-last spine, one level down to guild's own agents.

### From guild:whiteboard-skeptic

Round 1 of the Phase-4 whiteboard already burned through the testing-split, the tools-map totality, and the design-phase-domain-has-no-reviewer-cell traps — and those landed. That's genuine consensus that survived a round. I won't relitigate it. Phase 5 introduces three *new* hazards that none of those guards touch — the ones that live in the gap between "the manifest is consistent" and "the generated agent is the live, spawnable, equivalent thing."

**1. The deletion is a runtime-registration cliff, and the PLAN's exit order walks straight off it.** Agents are spawned by `subagent_type` *name* through Claude Code's registry (`guild-spawn` is explicit: it does not validate names, it trusts the runtime to resolve `subagent_type: evaluator-a11y` against the installed plugin copy). The whiteboard you're reading is running baked `guild:whiteboard-*` agents from the *installed marketplace copy*, not from `plugins/guild/agents/` source. So `guild generate` writes SOURCE files, and Phase 5's third exit bullet — "the old baked files are removed" — deletes the only copies the *current* installed plugin can resolve, while the generated replacements are *also* just source until a re-install registers them. There is a window where the repo has neither the baked originals (deleted) nor registered runtime agents (generated-but-not-installed), and in that window `guild-validate` and `guild-whiteboard` spawn names that resolve to nothing. Concrete remedy: Phase 5 must NOT couple "generate" and "delete baked" in one unit. Split — (a) `guild generate` lands + emits + tests prove parse/frontmatter; (b) re-install + a real spawn smoke (actually `Agent`-dispatch one generated reviewer and one generated planner, confirm a `VERDICT:` comes back through the aggregator); (c) only *then* delete the baked originals, as the close-the-loop unit. The deletion's exit criterion isn't "the generated files exist," it's "a re-installed plugin spawns the generated names successfully."

**2. The generated `name:` is the equivalence contract, and codegen has two ways to silently break it while staying parseable.** Fidelity isn't "does the frontmatter carry the right `tools:`" — that's nailed. It's "does the generated agent answer to the same `subagent_type` the callers hardcode." Every caller — `guild-validate`, `guild-whiteboard`, `derive-panel`, the loom panel overrides — names agents `evaluator-a11y`, `whiteboard-performance`, etc. But the generated files are composed from `(personality x domain x phase)`. If codegen names them by their axes (`reviewer-skeptic-a11y`) the files parse perfectly, carry correct frontmatter, *and* every existing caller's `subagent_type` resolves to nothing — a wrong-but-parseable agent in the purest form: green tests, dead panel. The second silent break: the manifest collapses all eight reviewer domains into *one* `[[combinations]]` row with a `domains` array; codegen must fan that single row into *eight* separately-named files — a naive "one file per row" reading emits one file for the whole array. Concrete remedy: make the name-mapping an *explicit, tested* part of codegen. The real-artifact regression is a *name-and-frontmatter equivalence snapshot*: for each baked file about to be deleted, assert a generated file exists whose `name:` equals the baked `name:` AND whose `tools:` set equals the baked `tools:` set (order-normalized). That test is the literal proof of "equivalent, not just well-formed," and doubles as Phase 7's green-light — you may delete `evaluator-a11y.md` exactly when a generated agent named `evaluator-a11y` with a matching tool set exists and spawns.

**3. Commit-the-generated-files is the lesser evil; the guard is a regen-clean CI check — gitignore reintroduces the exact failure class this project exists to kill.** Gitignore-and-generate-on-install means every consumer runs `guild generate` at install, which (a) widens the installed-vs-source split that caused hazard #1, (b) adds a failure mode where a stale/absent generate step means the panel silently runs nothing, (c) makes the spawnable runtime un-inspectable in the repo. Strictly worse for a single-operator substrate whose whole value is legible agent files. Commit them. The real objection — hand-edit drift from the manifest — is the same drift problem round-1 solved with a test, the same shape as loom's "state rides the feature commit." Concrete remedy: a `generate --check` mode (regen to temp, diff against committed, nonzero exit on drift) wired into CI, like a lockfile-freshness gate. Committed + drift-gated gives legible runtime agents, a reviewable diff when source changes, and a CI failure the moment someone hand-edits a generated file. Write a `DO NOT EDIT — generated by guild generate from panel.manifest.toml + tools-map.toml` banner into each file with the regen command — a committed generated file with no provenance banner is a trap for the maintainer who "fixes" it in place.

Two smaller notes the round-1 panel and existing test mostly cover: the consistency test guards `[phase.X]` totality but *not* domain-grant resolution or the reviewer-cell trap — it would pass a manifest listing a design-only domain in a reviewer row (the file exists, so the existence check is satisfied, while the agent would be hollow). Worth one assertion that no design-only domain (performance, substrate, composition, abstraction) appears in a reviewer combination. And the `whiteboard-skeptic` open question is the one genuinely-unresolved axis: a domain-agnostic skeptic doesn't fit the `(personality x domain x phase)` key, and Phase 5 will be tempted to emit a `skeptic x <each-domain> x planner` fan-out to force-fit it. Don't — emit one personality-led `whiteboard-skeptic` planner agent with no domain mode inlined, taking the brief's domain at dispatch. That's the one place the 3-axis model legitimately needs an escape hatch; naming it as such now beats discovering it as a wrong-but-parseable 8x fan-out later.

### From guild:whiteboard-testing-strategy

The temptation here is to reach for the expensive tier (spawn a real reviewer, watch it emit a verdict) when almost every risk in the exit criteria is deterministic and lives at the unit tier against the *real* generated artifact.

**The load-bearing testability property is determinism — design `generate` for it first.** Codegen that produces byte-identical output on repeat runs is the difference between "we can write a freshness test" and "every test is a fuzzy snapshot that rots." The tools-map fold already commits to normalized ordering, so the contract is deterministic by design. Pin it: a unit test that runs `generate` into a temp dir twice and asserts the two trees are byte-identical. That single test defends the whole class of "someone introduced a `Set` iteration-order dependency or a `Date.now()` in a header" regressions, and it's the precondition that makes the freshness check trustworthy.

**Tier + shape for the four exit sub-criteria — all four are cheap unit tests against real generated output, not fixtures.** The repo's origin lesson (fixtures pass green while the real shipped artifact is broken) bites hardest here, because `generate` *is* an author→consumer pipeline. Run the real `generate` against the real manifest + tools-map + fragments, into a temp dir, then assert on the emitted files:

- **(a) Generated files parse as valid agent markdown-with-frontmatter.** Generate to temp; for every emitted file split frontmatter, assert the YAML parses, assert required keys exist (`name`, `description`, `tools`, `model`, `maxTurns`). There's no compiler between a `.md` template concatenation and a broken agent file, so the test is the only catch.
- **(b) Synthesized `tools:` matches the tools-map fold.** The sharpest one. Don't snapshot the `tools:` line — *recompute the expected fold independently in the test* (`phase.base ∪ domain.grants`, deduped, normalized) and assert equality against what `generate` emitted. Two implementations agreeing is real signal; a snapshot is "it equals what it equalled last time." Cover three fold cases: phase-base-only (reviewer × naming → exactly `Read, Glob, Grep`), phase-base-plus-grant (reviewer × a11y picks up `test:a11y`), and the load-bearing negative — **a generated reviewer carries no `Write`/`Edit`** (the `writes = false` floor; its own test). Also assert the fail-loud contract: feed `generate` a manifest naming a phase with no `[phase.X]` row and assert it errors rather than emitting a permissive empty `tools:`.
- **(c) Inlining actually happened — zero dangling dispatch reads.** The baked `evaluator-a11y.md` literally instructs "read `evaluator-base.md`"; after inlining those dispatch-time read instructions must be gone. Assertion: no generated file contains a dispatch-time "read `<fragment>.md`" instruction, allowlisting the legitimate runtime reads an agent still makes (the artifact under review, `npm run` outputs). Pair with a positive assertion so it can't pass vacuously: a known sentinel from the base contract (e.g. the verdict-format block) is present inline in every generated reviewer.
- **(d) Generated reviewers emit a VERDICT parse-and-aggregate accepts.** People will want to spawn an agent — don't. The breakable thing is the output-contract *template* the agent is told to emit, a static string inherited from `reviewer.md`. Test it statically: assert every generated reviewer's inlined output contract contains the `VERDICT: approved`/`flagged` skeleton, then feed that skeleton through the actual `parse-and-aggregate` verb (you have the consumer in-process) and assert it parses. Mind the line-anchored gotcha: the parser anchors `VERDICT:` at line start, so assert the skeleton satisfies the line-anchored regex, not just that the substring appears.

**The panel smoke test: draw the boundary at the deterministic seam, keep the LLM out of CI.** "Confirms the collapsed panel still emits verdicts" decomposes into two halves. The deterministic half — does a generated reviewer's output template parse into a verdict — is (d), belongs in CI as a fast in-process integration test. The expensive half — does a live LLM agent produce a conformant verdict on a real diff — is a real LLM call: slow, nondeterministic, costs tokens, flakes. Make it a documented manual/operator smoke (one generated reviewer spawned against a tiny known-bad sample, eyeballed for `flagged`), run once at Phase 5 landing and recorded in the checkin, not on every push. Mirror `derive-panel.test.ts`, which tests deterministic panel *composition* in-process and never spawns the evaluators it names. Test the codegen output thoroughly and cheaply; smoke the live spawn thinly and manually.

**Determinism check vs freshness check — you want both, and the commit decision picks which is load-bearing.** Idempotency: `generate` twice → byte-identical (independent of the commit decision). Freshness: the committed generated files match what `generate` produces now (generate to temp, diff against committed tree, assert empty) — this only exists *if generated output is committed*. If committed, the freshness check is the headline test and the cheapest signal — it makes "did you re-run generate after editing a mode?" a CI failure instead of a review catch. My lean on testability grounds: **commit the generated files** — it buys the freshness test, makes `tools:` lines reviewable in PR diffs, and sidesteps the install-environment failure modes a generate-on-install step introduces.

**How the Phase-4 guards relate — keep both, they guard a different seam.** `flag-code-drift.test.ts` + `panel-manifest-consistency.test.ts` guard the **source** (do the modes retain the baked catalog; do the manifest's references resolve). Phase 5's tests guard the **output** (does codegen fold source into generated files correctly). Not redundant; Phase 5 doesn't supersede them. `panel-manifest-consistency.test.ts` is the *input precondition* for `generate` — keep it, it fails fast at the source layer with a sharper error than a codegen crash. `flag-code-drift.test.ts` guards a window that **closes in Phase 7**: it reads `agents/evaluator-*.md`/`whiteboard-*.md`, which Phase 7 deletes — after which it `ENOENT`s. Keep it unchanged through Phases 5-6 (still guarding live drift), and decide its Phase-7 fate explicitly: either it dies with the baked files, or it gets re-pointed to assert the flag codes survived into the *generated* output. The clean handoff: Phase 5's (c) inlining test can absorb the "flag codes present in generated output" assertion, so by Phase 7 the flag-code invariant has a home that doesn't depend on the baked files existing.

One precondition for the (b) test: the `whiteboard-skeptic` shape question (skeptic × planner, domain-agnostic) must be resolved before its test can be written — a domain-agnostic agent has no `[domain.X]` row to fold, so its expected `tools:` is phase-base-only and the test looks different from the domain-bound combinations. Whoever resolves the shape owns telling the test author which fold case it falls into.
