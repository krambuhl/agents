# Whiteboard: Design the shared PLAN.md parser lib (parsePlan) for the loom CLI

## Round 1

### From guild:whiteboard-substrate-engineer

The shape question that has to land before any type definitions get drawn: **`parsePlan()` is a pure read-derivation, and that's the invariant the whole lib hangs on.** It reads PLAN.md text, folds it into a tree, returns it ‚Äî no writes, no manifest coupling, no current-phase awareness. The Phase 1 GOAL says this explicitly ("side-effect-free‚Ä¶ does not write"), and that purity is what makes Phase 2 and Phase 3 safe to build on it. Keep `parsePlan(text: string): ParsedPlan` taking *text*, not a path ‚Äî let the caller own the `readFileSync` and the `LoomError('plan-not-found')` (mirroring `readManifest`'s split). A text-in/tree-out function is trivially unit-testable, fixture-friendly, and can't grow a hidden filesystem dependency. The moment it takes a path, it's two responsibilities and a harder test.

On the **tree shape (Q1)** ‚Äî this is the most consequential decision in the unit, because every downstream consumer reads against this shape and a reshape later is a hard migration across Phases 2/3/6. My read of the three fixtures: milestones (`### M1`) and phases (`#### Phase N`) are real and stable; deliverables are *not* a PLAN.md concept. The jelly fixture's own decision log says it outright ‚Äî "Tasks NOT declared in PLAN.md ‚Äî lead agent decomposes at `/goal`-time." So phases are the leaf of the parsed tree. The ev-loop-interactive prose that says "parse the phase's deliverables from PLAN.md" is parsing the `**Exit**:` bullets ‚Äî those *are* the deliverables operationally, but in the tree they should be named `exitCriteria`, not `deliverables`, because that's what the heading actually says and because conflating them invents a vocabulary the source doesn't have. Lead the tree with `{ milestones: Milestone[] }` where each milestone carries `phases: Phase[]`, and let phases be flat-addressable too (a `phasesByNumber` index or a `phases` flat accessor) ‚Äî because `ev-run` picks "the next actionable phase" by *number and dependency*, not by milestone, and forcing it to walk the milestone nesting to find phase 4 is orchestration-hostile. Two views over one parse: nested for humans, flat-by-number for the dependency resolver.

On **milestone-vs-phase nesting specifically**: don't make milestones mandatory. The linear-loom fixture has `## Phases` with flat `### Phase N` and *no* milestone layer at all; jelly and substrate-consolidation both nest. If the type makes `milestones` the only path to phases, the linear-loom shape forces a synthetic "M0" wrapper ‚Äî that's the family-inconsistency smell, inventing structure to satisfy a parser. Better: phases are first-class, milestones are an *optional grouping annotation* over them (`phase.milestone?: { id: string; name: string }`). One shape absorbs both fixtures without a fake node.

On **parser tolerance (Q2)** ‚Äî lean lenient-with-diagnostics, not throw-on-malformed, and here's the substrate reason, not just an ergonomics one: this parser runs against PLAN.md files that humans hand-edit and that `loom revise-plan` (Phase 3) rewrites. A strict parser that throws on the first unrecognized heading turns every malformed-but-mostly-fine plan into a hard CLI failure, and worse, makes Phase 3's atomic PLAN.md-rewrite fragile (one bad edit and the next parse throws). Return `{ plan: ParsedPlan; diagnostics: Diagnostic[] }` where diagnostics carry a stable `code` + line number ‚Äî and crucially, **reuse the `LoomError` code vocabulary as the diagnostic-code vocabulary** so a verb can decide blocking-vs-advisory the same way it does everywhere else. The distinction I'd draw: structural breakage that makes the tree meaningless (no phases found at all, a `**Depends on**` pointing at a phase number that doesn't exist) is a *diagnostic the caller can choose to treat as fatal*; cosmetic drift (an unrecognized `**Foo**:` block, a phase missing `**Goal**`) is advisory and the tree still returns. Let the verb own the throw decision; the lib reports.

On **dependency representation (Q3)** ‚Äî this is where the append-only/cross-reference instinct matters. `**Depends on**: Phase 1, Phase 2` is a *cross-reference into the same tree*, and the parser should resolve it to phase numbers, not leave it as a string the caller re-parses (re-parsing in `ev-run` prose is exactly the duplication Phase 1 exists to kill). Represent it as `dependsOn: number[]`, with `nothing` ‚Üí `[]` (empty array, not null ‚Äî `ev-run` wants to ask "are all deps satisfied" and `[].every()` is true by construction, which is the correct "no blockers" answer). Ranges like `Phases 1-6` from the M4 fixture must expand to `[1,2,3,4,5,6]` at parse time ‚Äî don't leak range-syntax into the consumer. And add a diagnostic for the dangling-reference case (depends-on a phase number with no matching `#### Phase N`); that's the one dependency error that silently breaks `ev-run`'s actionability math, so it's worth surfacing as a structural diagnostic rather than swallowing.

On **the `**Whiteboard**` override and `## Loop strategy` (Q1 tail)** ‚Äî these are real and load-bearing (the MEMORY note on the whiteboard-glob gap exists *because* the `**Whiteboard**` override is the documented workaround), so the parser must capture them ‚Äî but mind the level. `## Loop strategy` is plan-global; `**Whiteboard**` in the substrate-consolidation fixture is plan-global too, though in other plans it's appeared phase-local. Park `loopStrategy?: string` at the plan root and `whiteboard?: string` at *both* levels (plan-level default, phase-level override), with phase-level winning. Keep them as raw captured strings, not further-parsed ‚Äî the parser's job is to *find and hand off* these blocks, not to understand `engineers=guild:whiteboard-...` syntax. That syntax belongs to ev-loop, not loom's plan parser; parsing it here is orchestration leaking down into a CRUD-shaped lib.

On **section identification (Q5)** ‚Äî heading-driven, but anchor on the `#### Phase N ‚Äî Name` regex as the spine and treat heading *level* as advisory, not load-bearing. The fixtures already drift: substrate-consolidation uses `####` under `###` milestones; linear-loom uses bare `### Phase N` with no milestone. If the parser hard-requires `####`, linear-loom fails. Match on the `Phase N ‚Äî Name` *text pattern* regardless of `#`-count, and emit a diagnostic (not a throw) when the level is inconsistent with siblings. Same for `**Goal**:` / `**Exit**:` / `**Depends on**:` ‚Äî match the bold-label pattern, tolerate reordering, treat each as optional-with-diagnostic. Robustness to reorder + optional-section is what lets the same parser survive `loom revise-plan` rewrites without a version bump.

On **the regression corpus (Q4)** ‚Äî this is the lesson the predecessor project was born from, and the right framing is: **all three committed PLAN.md files are the corpus, checked in as real-artifact tests, not copied into synthetic fixtures.** The failure mode that bit `loom-pr-reconcile-verb` was a fixture diverging from the shipped artifact; the defense is to point the test directly at `projects/*/PLAN.md` on disk. Concretely ‚Äî a parametrized test that globs the real `projects/**/PLAN.md`, parses each, and asserts zero *structural* diagnostics (cosmetic ones are allowed and expected to differ across plans). Plus one golden-tree snapshot against substrate-consolidation's own PLAN.md (the richest, and the one the GOAL names explicitly), so a reshape of the tree shows up as a reviewable diff. Synthetic fixtures still earn their place ‚Äî but only for the *malformed* cases you can't get from real plans (the dangling-dependency, the missing-`**Goal**`, the heading-level drift). Real plans prove it works on what ships; synthetic fixtures prove the diagnostics fire. And the Node strip-only e2e smoke the PLAN demands applies here too: one `node parse-real-plan.ts` invocation outside vitest, because a parameter-property or `*/`-in-JSDoc slip in this lib gets masked by vitest exactly like the TOML parser risk.

What I'd want the next round to resolve, and what I defer: I'd want the panel to nail down **whether milestones are optional grouping or mandatory nesting** (Q1) before any types get written, because that's the one decision a Phase 2/3 reshape can't cheaply undo. On the markdown-AST-vs-hand-rolled question (the PLAN's "weigh against a dependency") I lean hand-rolled to match the rest of `lib/` and keep the zero-dep posture, but that's genuinely a cost/correctness tradeoff ‚Äî defer the measurement to `whiteboard-performance`. The exact diagnostic *taxonomy* (which codes are structural-fatal vs advisory) is a `whiteboard-testing-strategy` and contract-fit concern once the shape lands; I only assert that the codes should reuse the existing `LoomError` kebab-case vocabulary rather than inventing a parallel one.

Files I read that are load-bearing for these claims:
- `projects/2026-05-26-substrate-consolidation/PLAN.md` (richest fixture ‚Äî milestones, `#### Phase N`, exit bullets, `**Depends on**`, plan-level `## Loop strategy`)
- `projects/2026-05-25-jelly/PLAN.md` (decimal phases `1.1`/`1.2`, "Tasks NOT declared in PLAN.md" decision)
- `projects/2026-05-21-linear-loom/PLAN.md` (flat `### Phase N`, no milestone layer ‚Äî the shape that breaks mandatory-nesting)
- `plugins/loom/cli/lib/types.ts`, `project.ts`, `manifest.ts`, `errors.ts` (the typed-lib + `LoomError` + text-in/path-split patterns to mirror)

### From guild:whiteboard-testing-strategy

The good news is the repo already contains the answer to the scariest part of this contract, so the work is less "invent a strategy" and more "point the new lib at patterns that already exist here." Let me name each risk and the cheapest tier that defends it.

**The strip-only footgun is the one risk vitest structurally cannot catch ‚Äî and it's already solved in this tree.** `loom.test.ts` lines 181-212 run the entry point through `spawnSync('node', [LOOM_ENTRY, ...])`. That subprocess is the *only* tier that exercises Node's strip-only loader; vitest transforms the TS through its own pipeline and will happily run a `constructor(public x: T)` or a `*/`-in-JSDoc file that real `node` rejects with `ERR_UNSUPPORTED_TYPESCRIPT_SYNTAX`. So the contract for `parsePlan` needs exactly one such smoke: a test that shells `node` against a tiny harness which `import`s the lib and calls `parsePlan` on one real PLAN.md, asserting exit 0. It doesn't need to assert the parse *tree* ‚Äî that's the unit layer's job. It exists to defend one risk: "the lib file loads at all under the loader it ships on." One spawn, one assertion, done. Critically, that smoke must import the **lib module directly**, not go through a CLI verb ‚Äî a verb-level smoke would catch a verb's footgun but mask one living in `lib/plan.ts` if no verb imports it yet (Phase 1 ships the lib before the consumers). Defends against: a parameter-property or JSDoc-terminator that compiles green in vitest and explodes the first time loom runs under `node *.ts` in production.

**Tier split for the parser itself: this is overwhelmingly a pure-function unit problem, and that's a gift.** `parsePlan` reads text and returns a tree ‚Äî no FS writes, no git, no clock, no network. That means no mocks, no temp dirs, no `beforeEach`/`afterEach` cleanup, and inherently parallel-safe (the `plan.test.ts` dance with `mkdtempSync` exists only because that verb touches the filesystem; `parsePlan` shouldn't need a byte of it). Feed it a string, assert the tree. The one design constraint that protects this: keep `parsePlan(text: string)` taking text, not a path. If it grows a `parsePlanFile(path)` convenience wrapper, keep the file-read in a *separate* thin function so the core stays a string‚Üítree pure unit. Defends against: logic regressions in milestone/phase/exit/depends-on extraction, at the cheapest tier that gives full signal.

**On the fixture-vs-real-artifact question (your Q4) ‚Äî you need both, and they defend different risks. Don't let one masquerade as the other.** The split:

- *Synthetic fixtures* (small hand-written PLAN strings, ideally inline in the test or in `fixtures/plan-*.md`) are what the parser **asserts against**. `it('extracts exit criteria as a string array')` wants a 6-line plan where you know the exact expected tree. These are stable shapes the test owns; a real 148-line PLAN.md is a terrible assertion target because every plan edit would churn the test and you'd end up asserting `toBeTruthy()` instead of `toEqual([...])`. One concept per fixture: a depends-on fixture, a multi-milestone fixture, a missing-exit fixture.

- *Real committed PLAN.md files* are what the parser **survives**, not asserts against. This is the `loom-pr-reconcile-verb` lesson made concrete: fixtures passed green while the real shipped artifact was broken. The defense is a glob test ‚Äî `Glob('projects/**/PLAN.md')`, parse each, assert *structurally* (parse doesn't throw, returns ‚â•1 phase, every phase has a name). You're testing "the parser handles the shapes that actually exist in this repo," not "this specific plan has 7 phases." That assertion stays green as plans evolve and goes red the moment someone writes a real plan shape the parser chokes on. The three plans you handed me are perfect corpus seeds ‚Äî the substrate-consolidation one with `#### Phase N ‚Äî Name` + `M1` milestone headers, jelly with `#### Phase 1.1` dotted numbering *and* a duplicate `**Goal**:`/`**Goal (updated)**:`, linear-loom with `### Phase N ‚Äî Name` (three hashes, not four) and a prose `## Dependencies` section instead of per-phase `**Depends on**:`. That heterogeneity is the actual spec. The glob test is the one place the self-referential "including this project's own PLAN.md" exit criterion gets honored automatically ‚Äî it'll be in the glob the day it's committed.

One trap in the glob approach worth flagging now: a `Glob('projects/**/PLAN.md')` test is only as good as the corpus present at run time, and it couples the unit suite to repo layout. Keep it honest with a floor assertion (`expect(plans.length).toBeGreaterThan(2)`) so an empty glob can't pass as "all real plans parse." Defends against: the glob silently matching zero files and going green ‚Äî the exact false-confidence shape the predecessor project got burned by.

**On Q2 (strict vs lenient) ‚Äî the tolerance decision is a test-shape fork, so decide it before the contract is written, not during.** Look at the corpus divergence: linear-loom uses `### Phase` (3 hashes), the consolidation plan uses `#### Phase` (4). jelly has both `**Goal**:` and `**Goal (updated)**:` on the same phase. If `parsePlan` is *strict* (one canonical shape, throw on deviation), then the real-artifact glob test will go red on two of your three seed plans today ‚Äî which tells you strict is the wrong call unless you also normalize the corpus first. If it's *lenient* (extract what's there, tolerate heading-level variance and missing sections), the malformed-input test isn't "throws on bad input," it's "returns a partial tree with the missing pieces absent/empty." My lean, given the heterogeneity you're handing the parser: lenient extraction with a typed-but-optional shape (`exit?: string[]`, `dependsOn?: string[]`), and the malformed test asserts *graceful degradation* (`phase with no **Exit**: ‚Üí exit is undefined, not a thrown error`) rather than rejection. A planner-facing parser that throws on the first plan that omits a section is a parser that fights its own corpus. Reserve thrown errors for input that isn't a plan at all (no `#` headings whatsoever) ‚Äî that's the one genuinely-malformed case worth a `toThrow`. Defends against: a brittle parser that's correct on the canonical shape and useless on the four shapes real plans actually take.

**What I'd explicitly leave to CI, not tests.** The "no skill body re-implements PLAN.md parsing" exit criterion is a *grep/lint* assertion, not a unit test ‚Äî a test asserting the *absence* of a regex in two skill `.md` files is belt-and-suspenders that rots the moment a third skill appears. If you want to defend that invariant, it's a one-line `Grep` check in CI or a review-time eyeball, not a vitest case. Don't build the test that re-greps the skills; the type system and a code-review pass already carry that load.

**What I'd want resolved before the unit contract lands:** (1) the strict-vs-lenient call from Q2, because it's load-bearing on every parser test's shape and on whether the glob test is green or red on day one ‚Äî `whiteboard-substrate-engineer` should weigh in on whether a normalized PLAN.md heading convention is worth enforcing upstream (which would let the parser be stricter and the tests sharper); and (2) whether the markdown-AST dependency is in play, since a hand-rolled line scanner and an AST walker have different failure modes and the strip-only smoke gets more important, not less, if a dependency is added (the dep ships its own loader surface). The AST-vs-handrolled cost tradeoff is partly `whiteboard-performance`'s lane; I only care that whichever wins, the string‚Üítree purity and the one real-`node` smoke both survive the choice.

Relevant paths for the next unit contract:
- `plugins/loom/cli/loom.test.ts` (lines 181-212) ‚Äî the exact `spawnSync('node', ...)` smoke pattern to mirror for strip-only safety.
- `plugins/loom/cli/verbs/loom/plan.test.ts` ‚Äî the established pure-vs-FS test split; `parsePlan` lives entirely on the pure side and should *not* inherit the `mkdtempSync`/`afterEach` machinery.
- `plugins/loom/cli/lib/manifest.ts` ‚Äî the typed-lib + `LoomError`-coded-throw pattern the lenient-parser error path should follow for its one genuine-malformed case.
- `plugins/loom/cli/fixtures/` ‚Äî where synthetic `plan-*.md` assertion fixtures belong (alongside the existing JSON fixtures).
- The three real plans ‚Äî the real-artifact glob corpus; their heading-level and section-presence divergence *is* the parser's tolerance spec.

### From guild:whiteboard-skeptic

## Skeptic ‚Äî Phase 1 plan-parser premise pressure-test

I read all three named fixtures plus the archived `marketplace-portable-install/PLAN.md`, the two consumers (`ev-loop-interactive`, `ev-run`), and the lib pattern you're being asked to mirror (`manifest.ts`). Verdict up front: **the premise is sound, but the goal as written over-promises, and the divergence across real plans is larger than the brief's framing implies.** Three sharp risks, each with a concrete remedy.

### Risk 1 ‚Äî The em-dash is the parser's whole world, and three of four plans don't agree on heading level

I grepped the actual heading tokens across every PLAN.md in the repo. The structural divergence is not hypothetical:

- **Phase heading level drifts.** `substrate-consolidation` and `jelly` use `#### Phase` (nested under `### M1` milestones). `linear-loom` uses `### Phase` (no milestone layer at all ‚Äî it has `## Dependencies` where the others have milestones). The archived `marketplace-portable-install` has a flat `## Phases` section with *no per-phase `###`/`####` headings that this grep caught at all* ‚Äî its phases are structured differently again. So across 4 real plans you have **at least 3 distinct phase-heading shapes**, and milestone presence is 50/50.
- **Phase numbering is not integer.** `jelly` uses `Phase 1.1`, `Phase 2.2`, `Phase 3.1`. The others use `Phase 1`, `Phase 7`. A naive `Phase (\d+)` capture silently truncates `1.1` to `1` and collides `1.1`/`1.2`/`1.3` into one phase. The brief's example `**Depends on**: Phase 1, Phase 2` assumes integer IDs that two of three fixtures violate.
- **The em-dash is load-bearing and inconsistent.** `Phase 1 ‚Äî Name` uses U+2014 (em-dash). But `**Depends on**: Phases 1‚Äì6` in `substrate-consolidation` uses U+2013 (en-dash) for the *range*, and `Phases 2-8` in `linear-loom`'s status line uses an ASCII hyphen. If the parser keys on `‚Äî` it'll handle headings but choke on dependency ranges, and a human who types a hyphen instead of an em-dash in a heading (every non-Mac keyboard, every paste from Linear/GitHub) produces a phase the parser can't see.

That last one is the treadmill the brief asked about, and it's real: a contributor edits PLAN.md, types `Phase 8 - New thing` with a hyphen, and `parsePlan()` either drops the phase or throws. The failure is invisible until a downstream verb acts on a truncated tree.

**Remedy:** Make the heading regex tolerant by construction, and write the tolerance into the test fixtures, not just the prose. Concretely: match `^#{2,4}\s+Phase\s+([\d.]+)\s*[‚Äî‚Äì-]\s*(.+)$` ‚Äî accept 2‚Äì4 hashes, capture a dotted ID as a *string* (never coerce to int), and accept any of the three dashes. Add a fixture per shape: the integer-milestone shape (`substrate-consolidation`), the dotted-no-milestone-but-milestone-present shape (`jelly`), the no-milestone flat shape (`linear-loom`), and a deliberately-hyphenated adversarial fixture. If a `## Phases` / `## Milestones` section exists but yields **zero** phases, that's a structured error (`plan-no-phases-found`), not an empty array ‚Äî silent-empty is the failure mode that bites a downstream consumer.

### Risk 2 ‚Äî "No skill body re-implements PLAN.md parsing afterward" is an exit criterion the lib can't satisfy, because the consumers don't parse the same things

The brief's exit criterion is binary: *no skill body re-implements PLAN.md parsing.* But look at what the consumers actually read, because they don't read phases-and-dependencies ‚Äî they read **deliverables** and the **`**Whiteboard**:` override block**:

- `ev-loop-interactive` SKILL.md:157 ‚Äî *"Parse the phase's deliverables from PLAN.md. Each deliverable becomes one unit."* Deliverables are the load-bearing thing the loop consumes. They live as freeform bullets under `**Output**:` (linear-loom), `**Exit**:` (substrate/jelly), or unstructured prose ‚Äî there is **no consistent deliverables key** across the three fixtures. `linear-loom` Phase 3 has a `**Output**:` bullet list; `substrate-consolidation` has none, only `**Exit**:` criteria. The loop's "5 deliverables = 5 checkins" heuristic is reading *exit bullets as deliverables* in some plans and *output bullets* in others.
- `ev-loop-interactive` SKILL.md:104-113 ‚Äî the `**Whiteboard**:` override is a semicolon-delimited mini-DSL (`engineers=...; topic=...; rounds=...`), a completely different grammar from the phase tree.

If `parsePlan()` returns `{phases, exitCriteria, dependencies}` (the goal's stated shape) but the consumers need deliverables and the whiteboard block, then **the consumers keep their prose parsing** and the exit criterion is false on day one. You'll have shipped a typed tree that the two named consumers don't fully consume, which is the "gold-plated tree nobody asked for" failure.

**Remedy:** Before writing the lib, reconcile the return shape with what the consumers actually read. Either (a) widen `parsePlan()` to include `deliverables` (with an explicit, documented rule for *which* bullet block counts as deliverables ‚Äî and force the plans to converge on one key, probably `**Deliverables**:` or a documented `**Output**:`/`**Exit**:` precedence) and the parsed `whiteboard` override, or (b) scope the exit criterion honestly to "no skill re-implements *phase/dependency* parsing" and leave deliverable-reading as a named follow-up. Don't let the exit criterion claim a completeness the return type doesn't deliver. My recommendation is (a) but only after the panel agrees which bullet key is the deliverables key ‚Äî otherwise you're encoding today's accident as tomorrow's contract.

### Risk 3 ‚Äî This couples machine behavior to human prose *one phase before* you build the manifest that should hold the structured data

This is the one I'd most want the panel to sit with. Phase 2 of this *same plan* builds `manifest.toml` with a `[[phases]]` section. Phase 3 explicitly says revisions get *"machine record in the manifest, human rationale in the plan."* The whole project's thesis is **decision artifacts vs process artifacts** ‚Äî and a parser that derives machine-actionable phase/dependency structure from hand-edited markdown is the exact coupling the manifest is supposed to dissolve.

So there's a real ordering tension: Phase 1 invests in robustly parsing prose that Phase 2 may make authoritative-elsewhere. If `[[phases]]` in the manifest becomes the source of truth for phase identity and dependencies, then `parsePlan()`'s phase/dependency tree is either (a) redundant with the manifest, or (b) a validation cross-check against it ‚Äî both of which are *narrower* than "parse the full phase tree from prose." The risk is building a comprehensive prose parser in Phase 1 that Phase 2 demotes to a lint check.

This isn't an argument to skip Phase 1 ‚Äî the consumers need *something* today, and `parsePlan()` is the honest extraction of logic that's currently smeared across skill prose. But it changes what the *cheapest correct version* is.

**Remedy ‚Äî and this is my answer to the brief's "cheapest version" question:** Scope Phase 1 to **exactly what the consumers read today, no more**: phase headings (id + name + level-tolerant), the `**Depends on**:` edges, and the deliverable/exit bullets the loop turns into units. Treat exit-*criteria* as opaque strings, not a parsed sub-tree ‚Äî nobody consumes their internal structure, so parsing them into anything richer than `string[]` is speculative. Explicitly defer any "validate PLAN.md against manifest `[[phases]]`" logic to Phase 2 where the manifest exists, and **leave a named seam for it** (e.g. `parsePlan()` returns phase IDs as strings precisely so Phase 2 can join them against `[[phases]]` keys). Write a one-line invariant comment at the top of the lib: *"PLAN.md is the human-authored source; this parser is tolerant-read-only and never the authority on phase existence once manifest.toml ships (Phase 2)."* That comment is the exit hatch the next maintainer needs ‚Äî without it, someone in six months hardens this parser into a strict validator and breaks every plan that doesn't match the template.

### What I'm *not* flagging (clean passes)

- **Hand-rolled vs markdown-AST dependency:** hand-rolled is correct here. The structure is line-oriented (headings + `**Key**:` bullets), an AST gives you nothing a line scanner doesn't, and it violates the zero-dep lean. No concern.
- **Read-only / no-write:** correctly specified and matches the `manifest.ts` mirror (`readManifest` is pure read). No concern.
- **Node strip-only safety:** already a named hard constraint with a real-`node` smoke test in the plan's Verification section. The lib has no classes/constructors anyway (pure functions, like `manifest.ts`), so parameter-properties can't bite. Just don't put `*/` in a JSDoc example block that quotes a PLAN.md snippet ‚Äî that's the one place this lib could trip the footgun, since it'll be tempting to paste a `**Depends on**` example into a doc comment.

### Net

Premise: sound. Ship the parser. But the goal statement as written ("phases, exit criteria, and dependencies as a typed tree" + "no skill re-implements parsing") is **wider than the implementation can honestly satisfy and narrower than what the consumers actually read** ‚Äî it names exit-criteria (which nobody parses structurally) and omits deliverables and the whiteboard block (which the loop genuinely parses today). Fix the scope mismatch in the unit contracts before code: tolerant headings, string phase IDs, deliverables as a first-class field with a converged key, exit-criteria as opaque strings, and an explicit "Phase 2 manifest is the authority" seam. That's the cheapest version that delivers the real value and doesn't become the treadmill.
