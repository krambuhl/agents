# RESEARCH — guild used within Claude dynamic workflows

**Status:** Research complete — ready for `/loom-plan`
**Question:** How should guild be consumed inside Claude dynamic workflows, and which of three integration paths do we commit to?
**Scope:** The guild antagonist/whiteboard/write-capable agent family, the Workflow tool's primitives (`parallel()`, `pipeline()`, `agent({agentType})`, `agent({schema})`), the plugin/marketplace distribution boundary, and the half-built implementer/fixer write-capable expansion.

> Supersedes and extends `projects/WORKFLOWS-PARALLEL-ADOPTION.md` and `projects/WORKFLOWS-AGENT-POSTURES.md` (validated in PR #142, merged) with current-source verification as of 2026-05-31. Both prior dossiers carried two premises that have since gone stale; this dossier corrects them against live source rather than re-deriving the mechanics they already settled.

## Context

The goal is narrow and concrete: this repo wants guild to be *used within* Claude dynamic workflows — workflow scripts that fan out parallel agents and collect structured results — without guild having to become a workflow itself. Guild is a marketplace plugin; workflows are project- or user-local scripts. The whole question is where the seam sits.

Two prior dossiers did the foundational work and proved the core mechanics live. But the repo has moved underneath them since, and the user noticed the tell: guild now has write-capable `implementer`/`fixer` phases, but *only on the `css-architecture` domain*, while nearly every domain has `evaluator`/`whiteboard`. That asymmetry is not a bug and not an oversight — it is the **half-executed footprint of the prior posture research**. The `WORKFLOWS-AGENT-POSTURES.md` thread called for write-capable postures, someone seeded them on one proof domain on 2026-05-29 (right after the dossier), and deferred the expansion. This dossier picks that thread back up: it treats the write-capable expansion as in-scope, and it re-verifies the two now-stale premises before building a plan on top of them.

**Stale premises, corrected.** Six parallel readers verified the prior research against current source. Two premises were stale and are corrected here:

- **STALE PREMISE #1 (namespace forms) — corrected to FLAT, but the volatility is the durable fact.** The prior research validated three-segment agentTypes (`guild:generated:evaluator-*`, `guild:retained:evaluator-contract-fit`) and recorded that flat forms *failed* in demo run `wf_c3564478-de7`. The on-disk tree is now a **flat** `plugins/guild/agents/*.md` layout — the `retained/`, `generated/`, `personalities/` subdirs no longer exist. Agent frontmatter carries the bare `name:` (e.g. `evaluator-react`), with the `guild:` namespace applied by Claude Code at install. Operator memory (2026-05-31) *reports* the flat `guild:evaluator-*` form resolving and the prefixed forms failing — but this is operator recollection, not a fresh smoke; it agrees with the on-disk layout but does not substitute for the gate-(b) live workflow smoke. The correct durable takeaway is not "use the flat form" but **"never trust any hardcoded form; spawn-names are volatile across recompiles and must be read from the live registry at dispatch."** A subagent cannot confirm resolution; only a live workflow smoke can.

- **STALE PREMISE #2 (guild is purely read-only, write-capable phases missing) — OVERTURNED.** `WORKFLOWS-AGENT-POSTURES.md` claimed guild ships only read-only postures and that `implementer`/`fixer` are missing. ## What the prior research already settled

These are validated and should not be re-litigated by the plan:

- **The antagonist panel runs as a workflow.** `guild-validate-demo.js` spawns guild evaluators via `agent(packet, {agentType})` mapped over a `parallel()` fan-out, then mediates by spawning one thin Bash agent that runs `guild parse-and-aggregate` and returns only the verdict. This is real and end-to-end, not hypothetical (`.claude/workflows/guild-validate-demo.js:17-38`). Note the mediation call is plain `agent(prompt, {label, phase})` — **no `{schema}`** (verified: `grep 'schema' .claude/workflows/*.js` returns nothing). Schema was proven separately; see below.
- **The context win is measured at ~55% (floor) up to ~99%** on thin verdicts — the verbose evaluator bodies never return to the orchestrator; only sizes and the verdict do (`demo/DEMO-RESULTS.md`).
- **The boundary is a leaf.** A workflow is a pure function a loop *calls*: no PR border, no human pause, no git/manifest writes, no shell. The conductor (`ev-run`) and per-phase drivers (`ev-loop-*`) stay skills above the workflow line; only spawned agents have Bash (`WORKFLOWS-PARALLEL-ADOPTION.md:22-26`).
- **Mediation is Bash-wrapping the existing deterministic verb.** `guild parse-and-aggregate` does not move into JS; a thin agent shells it in repo-root with the plugin CLIs on PATH. Verified live — `printf '[]' | node plugins/guild/cli/guild.ts parse-and-aggregate` returns the locked six-field verdict (`verdict`, `blocking_findings`, `advisory_findings`, `cli_runs`, `conflicts`, `recusals`).
- **Schema-forced output works on prose-authored agents with no quality loss** — but this is proven only in *isolation* (probe `walh8d33b`, two evaluators), **not** wired into the same run as the parallel→mediate flow. Workflow mode does *not* require re-authoring evaluators; an `agent({schema})` call suffices for clean structured output. The catch is recorded below: schema captures structure, not policy.

## The three integration paths

### PATH A — Agents as workflow agentType

**What it is.** Workflow scripts call `agent(packet, {agentType: 'guild:evaluator-react'})` over a `parallel()` fan-out and Bash-wrap `guild parse-and-aggregate` to mediate. Guild stays a marketplace plugin, unchanged; workflows consume its agents. This is exactly what `guild-validate-demo.js` already prototypes.

**What's proven.** ~70% done — but the proof comes from **two separate runs**, and it matters which:
- The **parallel()+agentType+Bash-wrap-mediation flow is proven end-to-end** in `guild-validate-demo.js`, demo run `wf_c3564478-de7`: `parallel()` fan-out with concurrency cap (~16) and per-task `.catch` error containment; `agent({agentType})` resolution against the plugin-agent registry; Bash-wrapped verb mediation keeping `parse-and-aggregate` as the single source of truth. The measured ~55–99% context win is on this path.
- **`agent({schema})` for clean verdicts is proven only in isolation** (probe `walh8d33b`, two evaluators). It has **not** been exercised in the same run as the mediation path — the live end-to-end demo mediates with a plain `agent(prompt, {label, phase})` call, no `{schema}`. So "every primitive is demo-validated" is true *per primitive* but not *as one integrated flow*; wiring schema into the end-to-end pilot is an explicit P2 deliverable (below), not something already integrated.

**What it costs.** Three pieces of remaining work, all small and guild-ownable: (1) a real `guild map-agents`/normalization verb that turns bare logical names into the flat namespaced agentType (now closer to a prefix-prepend + existence-check + `-api`-strip than a subdir walk, since the layout flattened to 1:1), replacing the rotted `.mjs`; (2) the dual prose-vs-schema output contract, already implemented (`parse-and-aggregate` returns schema; evaluator bodies emit prose the aggregator parses); (3) wiring `agent({schema})` into the same run as the mediation path, since the two have only ever run apart.

**Against the bundling constraint.** Collides *least*. Guild ships through the marketplace unchanged; only the thin workflow `.js` lives project/user-local. The one irreducible gap is the live smoke: a subagent **cannot** confirm `guild:evaluator-react` resolves as a workflow agentType — only a real `wf_` run can.

### PATH B — Skills as workflow steps

**What it is.** Keep `/guild-validate` as the orchestration unit. Critically, the direction is **inverted** from the naive framing: a workflow *cannot* invoke a skill (workflows have no Skill tool or shell). The realized shape is a loop-skill (`ev-loop-confidence`) invoking a `guild-validate-panel` workflow *as a leaf*, with coordination staying in the skill layer (`WORKFLOWS-PARALLEL-ADOPTION.md:132-170`).

**What's proven.** This *is* the validated "Sketch 1" — and it is really **Path A wrapped in a skill**. The leaf it calls is the Path A workflow; the skill just owns composition and adds `Workflow` to its allowed-tools, awaiting completion. Same ~55% context floor.

**What it costs.** Almost nothing beyond Path A, because it *contains* Path A. Its only distinct virtue is keeping orchestration logic (panel composition, conflict handling) cohesively in the skill layer rather than split into the workflow script.

**Against the bundling constraint.** Sidesteps it cleanly — skills *are* a shippable component type. The skill degrades gracefully below the version floor (it works without dynamic workflows). But the leaf it calls still inherits Path A's agentType-resolution smoke dependency.

### PATH C — Reimplement guild on workflow primitives

**What it is.** `guild-spawn`'s parallel-spawn *becomes* the Workflow tool's `parallel()`/`pipeline()`; guild sheds its own spawn primitive and becomes workflow-native.

**What's proven.** Thin: that the move touches only the fan-out primitive — aggregation still stays a Bash-wrapped verb either way. That's the whole demonstrable upside, and it is small. The *genuine long-term* case for C is broader than what's proven: if a `workflows/` plugin component type ever lands, guild's primitives could ride it and the whole family (`guild-spawn`/`guild-validate`/`guild-whiteboard` plus `griot`/`loom`/`ev` consumers) could unify on a single workflow-native fan-out primitive instead of guild carrying its own.

**What it costs.** The most architectural change by far. It strips guild of its shippable parallel primitive, forces three skills (`guild-spawn`, `guild-validate`, `guild-whiteboard`) to become workflow-aware, and the one-level-nesting cap dissolves the family layering (a `guild-validate-panel` workflow could no longer be called by a `griot-judge-panel` workflow — the caller would have to be a skill). It would also have to preserve-or-rebuild the substantial, freshly-built matrix/codegen/cache machinery (`axes.toml` + fusion + `.cache.toml` provenance + drift-check) on workflow primitives.

**Against the bundling constraint.** Collides *hardest*. With no `workflows/` plugin component type, `guild-spawn`-as-workflow **cannot ship via the marketplace at all** — you would make guild's core function un-shippable through the family it belongs to. Disqualified on the primary gate. And even granting its genuine upside, C is a **future-option, not a now-path**: it is strictly dominated by Path A today, because the schema probe already showed guild's agents need *zero* re-authoring to work in workflows. So pre-component-type, C buys nothing Path A doesn't already get for free, while costing the most. The unification prize only becomes reachable *after* a `workflows/` component type exists — at which point C can be reconsidered as a consolidation, not adopted now as the integration path. Until then it is strictly more expensive for strictly no incremental capability.

## Recommendation

**Commit to PATH A**, with the orchestration optionally wrapped as PATH B where a loop-skill wants to own composition. They are not rivals — B is A wrapped in a skill, and the pilot can ship as a repo-local workflow that a loop-skill calls as a leaf.

The evidence is one-directional. Path A is the lightest path, already ~70% proven, and collides least with the one hard constraint, because guild ships through the marketplace unchanged while only the thin workflow leaf lives repo-local. Every primitive it needs is demo-validated *in isolation*, the core parallel→mediate flow is proven end-to-end, and the context win is measured on it — the one remaining integration step (schema in the same run as mediation) is a named P2 deliverable, not a discovered unknown. Path B adds nothing Path A doesn't already contain except a cleaner home for coordination logic — worth taking when a loop drives the panel, not worth treating as a separate path. Path C is ruled out by the primary gate (plugins can't bundle workflows); its real long-term prize — family-wide primitive unification — only unlocks *after* a `workflows/` component type lands, and pre-component-type it is strictly dominated by Path A since the schema probe already gave us zero-re-authoring for free. Do not pursue C until that component type exists.

Two non-negotiable gates on the recommendation: **(a)** build a guild-owned normalization verb (`guild map-agents`: logical name → flat namespaced agentType, with `-api`-suffix normalization baked in) so workflow scripts never hardcode volatile spawn-strings and never break silently on a recompile; **(b)** a mandatory **live workflow smoke** confirming the flat `guild:evaluator-*` form resolves as a workflow agentType before any production claim — a subagent cannot settle this, and operator memory corroborating it is recollection, not a smoke. The write-capable asymmetry is explicitly a **non-blocker** for Path A: implementer/fixer are just more agentTypes a workflow spawns the same way, and the css-architecture-only seeding is a deferred-expansion wave, not a gate on shipping the read-only panel.

## The write-capable expansion (in scope)

This is the asymmetry close — picking up the half-executed posture thread.

**Which domains get implementer/fixer next, and in what order.** The safe-to-write set tracks whether a domain owns a *mechanical, verifiable artifact* with a **runnable verify grant in `axes.toml`** (a runnable lint/build/test at the reviewer/implementer phase). This gate is load-bearing — without a runnable verify grant the reviewer phase cannot gate the write, so the domain is *not* Wave-1 safe regardless of how mechanical its transform is. The css-architecture proof domain carries such a grant at `axes.toml:113-121` (`writes = true`, `Edit`/`Write`/`Bash` granted at the write phases). **Data caveat — verify before ordering:** the wave assignments below assume each named domain carries an *equivalent* runnable verify grant at its reviewer/implementer phase in `axes.toml`. That has been confirmed only for css-architecture. For `tokens`, the test domains, and `nextjs`, the grant line must be cited in data before the wave is committed; **if the grant is absent, the domain is "safe ONCE a verify grant is added," and adding that grant becomes an explicit prerequisite step before the wave** — not a Wave-1 entry. Sequence by the three-phase risk pattern:

- **Wave 1 (mechanical, strong verify) — *gated on confirming the grant*:** `tokens` (literal→token is mechanical). The cleanest first expansion on transform mechanics — but it is Wave-1 *only if* `axes.toml` shows `tokens` carrying a runnable lint/build verify grant at its reviewer/implementer phase the way css-architecture does at `:113-121`. If that grant line is not present, add it first (prerequisite), then admit `tokens` to Wave 1.
- **Wave 2 (contract-shaped, runnable verify) — *same grant check*:** `test-unit`, `test-integration` (write a test to a contract, runnable verification), `nextjs` (`lint:nextjs` + build gate). Each is Wave-2 only with its runnable verify grant cited in `axes.toml`; otherwise the grant is the gating prerequisite.
- **Hold for human-paired work (judgment-heavy):** `naming` (renames ripple; semantic judgment a pragmatist write-agent will guess wrong), `a11y` (correctness needs real AT/markup judgment), `react` (API shape is high-craft, explicitly interactive-pairing per `CLAUDE.md`).
- **Cannot get a write phase without first getting a reviewer:** the planner-only domains (`abstraction`, `composition`, `performance`, `substrate`) declare `phases = ["planner"]` only — no reviewer to gate a write, no verify grant. Whether they should *ever* get an implementer (vs. being inherently design-exploration domains) is an open question worth recording as an explicit matrix invariant.

**The shape: worktree isolation + single-writer-per-task.** The Category-3 finding holds — *parallelize the thinking, serialize the write*. A workflow runs `parallel()` over per-task `pipeline()`s; each task is one implementer writing only its own task's files (single-writer-per-task by construction), with `isolation:'worktree'` giving each parallel task its own git worktree so N tasks never contend. Within a task the implementer → reviewer → fixer legs are sequential (fixer runs after the reviewer's verdict), so they never co-write. The dossier's "confirm no parallel writers land on one file within a task" follow-up is satisfied by this sequential-pipeline + per-task-worktree shape. (Note: `isolation`/`worktree` keys are *not* exercised in either current demo — the demos hand-sequence phases with `phase()` + `await` — so the per-task write pipeline is newly buildable but unproven live; it is the natural next smoke after the read-only panel.)

**The acceptEdits permission note.** Workflow agents always run in `acceptEdits` and inherit the tool allowlist regardless of session mode. For write-capable guild agents that is the *point* (`Edit`/`Write` are granted in the phase base), not a risk — the guardrail is worktree isolation + the single-unit scope discipline baked into the implementer/fixer prose, not the permission mode. The real caveat: `acceptEdits` means **no human edit-confirmation inside the workflow**, so the reviewer phase is the *only* gate. A domain whose verification is weak (no runnable test/verify grant) should not get a write phase. This is the load-bearing reason the "safe-next" set above is gated on the presence of a runnable verify grant in `axes.toml`.

**Output-contract schemas.** Both are already sketched in prose and need light formalization into structured output:
- **Implementer** returns `{ artifact (files touched), description, verification-evidence (lint/build/git output), corrections (contract deviations) }` and **no verdict**.
- **Fixer** returns `{ corrected-artifact, fix→finding mapping, re-verification-evidence, corrections }` and **no verdict**.
Both deliberately withhold self-verdict — the reviewer phase gates. This pipes cleanly: implementer-output → reviewer-input → fixer-input.

**Does one action-fragment template suffice?** Yes. The implementer/fixer phase fragments are domain-agnostic and already authored; expansion to a new domain D is a **data edit, not new code**: (a) add `implementer`/`fixer` to D's `phases` in `axes.toml`, (b) add D to the `implementer-default`/`fixer-default` recipe domains, (c) confirm D carries a runnable verify grant at its reviewer/implementer phase (add one if absent), (d) re-run `/guild-compile`. The one real prerequisite before fusing across many domains: add a **`fixer` guidance block + fixer/implementer to the role-mapping list in `fusion-prompt.md`** so multi-domain fusion is deterministic (currently fixer leans on the implementer branch and generalizes — fine for one cell, non-deterministic at scale). Recommend also re-introducing the deferred `validate.ts` `writes=true` coherence lint before opening writes to many domains.

## Constraints & adoption gates

The honest blocker list. Path-determining gates are marked.

- **PRIMARY, path-determining — plugins cannot bundle workflows.** There is no `workflows/` plugin component type. `marketplace.json` declares only `name/source/description/dependencies` per plugin; a repo-wide grep returns nothing. Workflows load only from project `.claude/workflows/` or user `~/.claude/workflows/`. This **rules out Path C** outright (guild's core would become un-shippable) and means even under Path A the consuming workflow `.js` cannot ride the marketplace — every consumer hand-copies it. This is the seam most likely to drift, and the reason the recommendation's deliverable is a *repo-local pilot*, not a distributed component.
- **Version floor (inherited).** Dynamic workflows require Claude Code v2.1.154+, a paid plan, and can be disabled org-wide. Any consumer depending on a workflow inherits that floor — so the workflow-delivered panel **cannot be the only path**; the `/guild-validate` skill must remain the floor-independent fallback. (This is why B-wraps-A matters: the skill degrades gracefully.)
- **Namespace coupling (path-determining for A, live-smoke-only).** The workflow agentType registry is a *different, volatile* namespace from the loop's Agent/Skill registry, and the resolving form has rotted once already (three-segment → flat). The durable fact is the volatility itself. **Mitigation:** the `guild map-agents` verb so scripts never hardcode the form. **Unverifiable by subagent:** which form resolves — flag for live smoke. Operator memory agreeing with the flat form is corroboration, not confirmation.
- **Dual output contract (soft gate, not a blocker) — preserve the POLICY, not the verb.** `parse-and-aggregate.ts` carries real policy that must not vanish into a naive JS path: VERDICT-line regex, Reasons/Advisory/Remedy block slicing, BLOCKING/ADVISORY severity routing, code-prefix extraction (`criterion-unmet` default), recusal parsing, remedy-to-reason pairing, and verdict precedence (`flagged-conflict > flagged > approved`). Schema captures *structure*, not this *policy* — and the probe confirms the gap concretely: `DEMO-RESULTS.md:151` records `schema_native_merge_worked: true`, where a trivial JS merge of the structured outputs replaced parse-and-aggregate *for basic aggregation only*, with no Bash-wrap verb on the schema path. So on the schema path the verb does **not** stay the mediator — which means the load-bearing thing to preserve is the *policy*, not the verb. Safe **provided the parse-and-aggregate POLICY (severity routing, verdict precedence, conflict detection, recusal parsing) is preserved — either by keeping the Bash-wrapped verb as mediator (prose path) OR by porting that policy into the schema/JS-merge (schema path). The basic JS merge alone is insufficient; it drops the policy.** The recommendation preserves the policy on the prose path by default and makes porting-or-keeping it an explicit acceptance criterion on the schema-wiring P2 deliverable.
- **derive-panel silent-fallback bug (low, dormant).** `loadSpec()` catches any read/parse failure and returns hardcoded `FALLBACK_RULES` with an in-band `warning` string rather than failing loud (`derive-panel.ts:203-224`). Today the spec parses cleanly (no warning observed live), but a future spec edit that breaks parsing would silently swap rule sets if the caller ignores the `warning` field. Also unfixed: consumer-project spec resolution (running `derive-panel` inside an installed plugin where `PANEL-COMPOSITION.md` lives at the plugin's own `docs/`) is cwd-relative only. (Related reproducibility note: `derive-panel` with no `--files`/stdin baselines to a single evaluator — pass files to see the full panel.)
- **Documentation drift (low, cleanup).** `marketplace.json` and `plugins/guild/CLAUDE.md` still advertise `generator-*` agents that no longer exist (zero `generator-*.md` on disk; the implementer fragment states generator behavior is now subsumed). The manifest misdescribes the inventory; worth correcting to `implementer-*`/`fixer-*` as part of adoption work.
- **`fusion-prompt.md` missing fixer branch (medium, prerequisite for the expansion wave only).** Detailed above; not a gate on the read-only panel.
