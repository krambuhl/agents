# Whiteboard: jelly-run substrate — orchestration skills wrapping /goal with operator-paired grill-gates

## Round 1

### From whiteboard-substrate-engineer

The substrate-shape question hiding under all three skills is the same one: **where does the durable state live, and who owns the write path to it?** jelly-loom already answered this for the project lifecycle — files in `projects/<slug>/`, a write-once `manifest.toml`, git as the audit trail, no `events.jsonl`. jelly-run's three skills are pure orchestration; they should write *zero* substrate state of their own and instead drive existing CLI verbs and git/PR primitives. The smell to watch for is any of these skills reaching for its own bookkeeping file (a `.jelly-run-state.json`, a "last grilled fields" cache, a feedback-round ledger). If a skill needs to remember something across turns, that's the design telling you the state belongs in a durable surface jelly-loom already owns — most likely the PR itself, or git. Press hard here: jelly-run is composition, jelly-loom is CRUD, and the boundary stays clean only if jelly-run owns no files.

**On (a), confidence-scoring — the over-confident-skip is a real silent-failure mode, and the remedy is structural, not better heuristics.** A confidence threshold that gates whether a grill fires is exactly the "idempotency by reviewer-vigilance" antipattern applied to operator attention: the safety depends on the heuristic being right, which it won't always be. Don't try to make the classifier better — make the skip *visible and reversible*. The final preview (already in the decision log) must render every field with its confidence and its source-of-derivation, including the silently-filled high-confidence ones, so the operator can see what the substrate decided *not* to ask about. That turns "skip" from a hidden action into a reviewable one. Concretely for the unit contract: the preview is not "here are the 2 fields I'm unsure about" — it's "here are all N fields, 2 flagged, edit any." The grill narrows attention; it never narrows the surface. That preserves yield-to-operator by construction rather than by the classifier behaving.

**On (b), `/goal` coupling — this is a schema-evolution problem wearing an API costume, and it wants an adapter seam.** `/goal`'s preamble shape and yield signal are an external schema jelly doesn't control, read by every `/jelly-run` invocation forever. The blast radius if Anthropic changes the preamble format is "every jelly project at once," which is the worst place for cost to live. Isolate it: the preamble *composition* and the yield *detection* should be two named, single-purpose functions (or a tiny `goal-adapter` module shared across the skills), not inline prose smeared through three skill bodies. Then a `/goal` shape change is a one-file edit with a version-pin note, not a three-skill archaeology dig. Pair it with the `claude --version` preflight the PLAN already names, and make the preflight *fail loud* on an unknown version rather than proceeding on a guess — a silently-wrong preamble is worse than a refusal.

**On (e), idempotency — this is where I'd push back hardest, because re-invocation is the *common* path here, not the edge case.** `/jelly-pr-feedback <PR#>` is explicitly operator-re-invoked after each comment round (per the decisions log), so "run twice" isn't a network-death recovery scenario — it's Tuesday. Two questions the contract must answer: (1) If the operator runs `/jelly-pr-feedback 64` twice on the same comment set, does it dispatch the fixers twice? It must not — and the natural idempotency key is the PR's own comment-resolved/reaction state, which is durable and append-friendly on GitHub's side. Mark handled comments (resolve the thread, or react) so a re-run reads "already addressed" from the PR rather than from any local memory. That keeps jelly-run file-less *and* idempotent by construction. (2) `/jelly-pr` opening a PR twice: gate on "does an open PR for this branch already exist," same as jelly-loom's `isCommitted` collision check gates research/plan. The pattern's already in the family (`git.ts` line 25); reuse the instinct.

One family-shape note for **(d)**: jelly-loom verbs are named bare-imperative (`research`, `plan`, `revise`, `adr`) and the MCP tools mirror them 1:1. The jelly-run skills are `/jelly-run`, `/jelly-pr`, `/jelly-pr-feedback` — that's consistent (operator-facing slash commands, `jelly-` prefixed). But note these are *skills*, not CLI verbs, and deliberately so: they orchestrate, they don't CRUD, so they correctly do *not* join the `jelly <verb>` MCP surface. Keep it that way. If a future impulse says "expose `/jelly-pr` as `mcp__jelly__pr`," that's orchestration leaking into the CRUD layer — resist it.

What I'd want resolved by the next round: the exact idempotency key for `/jelly-pr-feedback` (GitHub thread-resolution vs. comment reactions vs. a sentinel comment) — that choice is load-bearing for both idempotency and the visible-turn protocol, and it's the one place jelly-run touches durable state it doesn't own. I'd defer to testing-strategy on *how* to test the lossy-subagent-return boundary (point c), but flag the shape consequence: because subagent returns are lossy at the parent, the fixers' output must land in a surface the skill can re-read (the working tree / a commit / the PR), not be trusted from the Agent-tool result text — which, conveniently, is the same file-less-but-durable discipline idempotency already forces. The two constraints point the same direction.

Relevant files:

- `plugins/jelly-loom/cli/lib/git.ts` (the `isCommitted` collision-check pattern to reuse for PR-open idempotency)

- `plugins/jelly-loom/mcp/server.ts` (the CRUD MCP surface jelly-run skills should stay off of)

### From whiteboard-skeptic

The five tensions the brief names are real, but they don't all carry the same weight. Two of them (confidence-scoring and `/goal`-coupling) are load-bearing risks that will shape the unit contracts; the other three are mostly "do the obvious thing and write it down." Let me rank by where the design can actually break.

**(a) The confidence heuristic is the one that bites silently — and the named mitigation doesn't cover the failure mode.** PLAN.md line 113/167 says the mitigation for over-confident classifications is "always show final preview before PR opens." But a preview only catches over-confidence if the operator *reads it as carefully as a grill*. The whole point of draft-then-grill is to reduce operator attention on high-confidence fields — so the preview is exactly where a wrong high-confidence classification hides, because it's wearing a "we're sure about this" badge that trains the operator to skim. That's the compounding version of the bug: the mechanism that saves attention is the mechanism that buries the error. Concrete remedy: make confidence *legible and asymmetric* in the preview. Render high-confidence fields with their derivation receipt inline ("Risk level: low — derived from: no migration, no new deps, 1 file"), not just the value. And bias the heuristic toward false-negatives: when a field's confidence is *near* the grill threshold, grill anyway. A needless grill costs one operator turn; a skipped grill on a wrong `Rollout: revert` or `Risk: low` field ships a mis-described PR. The asymmetry of cost should be baked into the threshold, not left to a uniform cutoff. I'd want the unit contract to name the threshold *and* the tie-break direction explicitly, not just "scores per field."

**(b) `/goal` coupling: isolate the seam, and make the preflight check load-bearing, not advisory.** The real blast radius isn't "the preamble format changed" — it's that `/jelly-run` composes the preamble *and* depends on the yield shape to know when to auto-chain `/jelly-pr`. That's two distinct coupling points to one external API, and they fail differently: a preamble-shape change degrades quietly (goal runs with a malformed preamble, produces worse work), while a yield-shape change fails loud (the auto-chain never fires, or fires on the wrong signal). Don't treat them as one risk. Concrete remedy: put both behind a single `goal-adapter` module (one file, the only thing in jelly-run that knows `/goal`'s surface) with a `compatibleVersions` constant and a real preflight that *refuses to run* on an unrecognized `claude --version`, rather than warning and proceeding. The CLAUDE.md memory already has scar tissue here — `[Substrate-design principles]` says yield-to-operator is non-negotiable; a silent yield-detection failure violates that by stranding the work without yielding. The preflight gate is how you keep an upstream change from turning into a silent protocol violation.

**(c) Testing: stop trying to test through `/goal` — test the seams on either side of it.** The known finding (subagent returns are lossy at the parent boundary) isn't a testing obstacle to work around; it's telling you where the test boundary *is*. You can't assert on what the dispatched implementer "returned" — so don't. Assert on the side-channel jelly-loom already established: the CLI emits JSON on stdout, writes files, makes git commits. The testable units of jelly-run are pure functions with no `/goal` in them: preamble *composition* (PLAN.md + git state → preamble string), comment *classification* (comment text → fixed/ambiguous/stale/discussion-only), confidence *derivation* (diff + PLAN → per-field score), remedy *aggregation*. Each is input→output, unit-testable exactly like `appendRevisionLogEntry` and `nextAdrNumber` already are. The orchestration glue (`/goal` invocation, subagent dispatch) gets a thin adapter that's *mocked* in tests and *smoke-tested* manually in the dogfood — same posture as `GitRunner` injection in `git.ts`. The contract should say: classification and confidence-scoring ship as exported pure functions in their own files, precisely so they're testable without `/goal` in the loop. If a function needs `/goal` to test, it's carrying too much.

**(e) Idempotency is underspecified and it's the gap most likely to corrupt real state.** jelly-loom is write-once (manifest) and append-only (ADR log, revision log). jelly-run breaks that posture: it's interactive, re-invokable, and the named feedback trigger is *explicit operator re-invocation* (`/jelly-run <slug> "address feedback on #N"`). So what happens on the second invocation? The operator runs `/jelly-pr-feedback #42`, an implementer dispatch half-completes (timeout, error, operator Ctrl-C), and they re-run it. Does the substrate re-dispatch the already-fixed comments? Re-open a PR that's open? Append a duplicate remedy? Append-only is safe for an audit log because duplicates are inert; it is *not* safe for an action log where the action is "dispatch a fixer." Concrete remedy: the feedback flow needs a comment-resolution ledger keyed on PR comment ID, and classification has to read it — a comment already marked resolved/dispatched is skipped on re-run. This is the append-only-vs-write-once question the brief asks, and the answer is "neither, exactly" — it's append-only-with-dedupe-key. Name the key (GitHub comment ID) in the contract. Without it, the turn-based protocol loses its most important property: that re-entering a turn is safe.

**(d) Family-shape — the one place I'd push back on the consensus.** The sibling shape is clean: jelly-loom is *CLI verbs + an MCP wrapper over the same verbs*, all logic in testable TS, the skill/MCP layer thin. jelly-run is specified as *three skills* — markdown SKILL.md bodies. That's a different substrate (prose instructions to the agent vs. executable TS), and it's where the family quietly forks. The risk: classification and confidence-scoring logic ends up *living in the skill prose* ("classify each comment as...") rather than in a CLI verb, which makes (c) untestable and (a) un-auditable. I'd argue the contract should push the deterministic core down into jelly-loom-style CLI verbs (`jelly classify-comments`, `jelly compose-preamble`, `jelly score-pr-fields`) that the skills *call*, keeping the skill bodies thin orchestration glue exactly like loom's skills call the loom CLI. That keeps the whole family on one shape and makes the hard-to-test parts testable. If the panel's consensus is "three skills, logic in the prose," that's the false-consensus I'd want pressure-tested in round 2 — it reads simpler today and compounds into an untestable, unauditable core tomorrow.

What I'd want resolved before contracts: the confidence threshold + tie-break direction (a), the dedupe key for re-invocation (e), and whether classification/scoring live in CLI verbs or skill prose (d) — that last one decides whether (c) is even achievable. The `/goal` adapter seam (b) I'd defer to whiteboard-react-architect / substrate-engineer if they're framing the module boundaries; my flag there is narrow — make the preflight *gate*, not *warn*.

Relevant files: `projects/2026-05-25-jelly/PLAN.md` (Phase 2.1, ~line 97; decisions log 167-171), `plugins/jelly-loom/cli/lib/git.ts` (the `GitRunner` injection pattern to mirror for the `/goal` adapter), `plugins/jelly-loom/cli/verbs/adr.ts` (`nextAdrNumber`, the append-with-gap precedent for the dedupe-key question), `plugins/jelly-loom/cli/lib/manifest.ts` (write-once posture jelly-run departs from).

### From whiteboard-testing-strategy

The jelly-loom CLI already shows the right instinct for this family: pull the logic into pure functions (`appendRevisionLogEntry`, `resolveProject`) and inject the boundary (`GitRunner`, `CliContext`) so the tests never touch real git, FS, or the MCP transport — see `plugins/jelly-loom/cli/verbs/revise.test.ts` and `plugins/jelly-loom/mcp/server.test.ts`. The jelly-run skills face the identical problem one layer up, and the answer is the same shape: the testable substance is *not* the skill body. It's the pure derivation functions the skill body calls.

**Tier choice — extract the deciding logic out of the skill prose, unit-test that.** A skill that "wraps `/goal` and dispatches subagents" is mostly orchestration glue, and glue is the wrong thing to aim a test at. But three pieces of real logic hide inside these skills, and each is a pure function begging to be lifted into `cli/lib/` next to the loom helpers:

- *preamble composition* (`composePreamble(plan, gitState, phase) → string`) — pure string-building from PLAN.md + git state. Unit-test it: feed a fixture-shaped plan + git snapshot, assert the preamble contains the phase exit criteria and does NOT contain "open PR" (that's the load-bearing invariant from Decision 166 — a unit test that asserts the *absence* of the PR instruction is exactly the "test what you're afraid will break" defense).
- *confidence scoring* (`scoreField(field, plan, diff) → {value, confidence}`) — pure. This is tension (a) and it deserves the sharpest tests; see below.
- *comment classification* (`classifyComment(comment, diff) → 'fixed-intent'|'ambiguous'|'stale'|'discussion-only'`) — pure. Unit-test the classifier against a table of comment fixtures with known-correct labels.

If those three are pure functions with injected inputs, the skill body shrinks to "call function, render result, gate on operator" — and the part that *can't* be unit-tested (the actual `/goal` invocation, the actual subagent dispatch) is now thin enough that it doesn't need to be.

**Tension (a) — the over-confident classifier is a calibration risk, and the defense is an asymmetry test, not a coverage target.** The danger isn't "the score is wrong" — it's "the score is *confidently* wrong and silently skips a grill." The bug you're afraid of is a false-high-confidence on a field that needed operator eyes. So the test contract should defend the asymmetry directly: build a fixture table of `(field, plan, diff) → expected confidence band`, and assert that the cases you've *designed to be ambiguous* (diff touches a file PLAN.md never mentions; PR archetype is genuinely uncertain between bug-fix and refactor) land below the grill threshold. One concept per test: `it('scores motivation low-confidence when the diff spans files outside the phase scope')`. Don't test "scoring works" — test "scoring refuses to be confident when it shouldn't be." And keep the threshold itself a single named constant the test imports, so calibration drift is one assertion away from being caught. The final-preview gate (Decision 167) is the backstop, but a backstop you can't unit-test isn't a defense you can trust — make the scorer's humility the thing under test.

**Tension (c) — the lossy subagent boundary means you test the dispatch *plan*, not the dispatch.** This is the known jelly finding (subagent return values are lossy at the parent boundary), and it's a tier-choice clarifier, not an obstacle. You cannot reliably integration-test "dispatch a real implementer agent and read its result" — the boundary eats the result. So don't try. Split it:

- *Unit tier*: `buildDispatchTasks(classifiedComments) → Task[]` is pure. Assert that fixed-intent comments produce tasks, ambiguous/discussion-only do NOT (the yield invariant, Decision 170 — the substrate never auto-resolves ambiguous, so a unit test that asserts `classifyComment('ambiguous') ⇒ zero dispatched tasks` is the single most important test in this plugin).
- *Side-channel tier*: per the jelly memory finding, the parent can't read subagent returns, so dispatch must write to a side-channel (hook log / file / CLI stdout parse). Test *that* — assert the dispatch writes the expected task records to the side-channel, the same way `server.test.ts` asserts `git.commits` on the fake runner. The fake-`GitRunner` pattern generalizes: inject a `fakeDispatcher` that records calls instead of spawning, assert on the recorded calls.
- *What you explicitly don't test*: the real `/goal` round-trip and real agent execution. That's M3 dogfood (Phase 3.1) — a manual e2e exercise, not an automated suite. Name that gap in the unit contract so nobody tries to write a flaky `/goal`-spawning test.

**Tension (b) — isolate the `/goal` surface behind one adapter so the blast radius is one file.** If preamble/yield shape changes upstream, you want exactly one module to edit. Define a `GoalAdapter` seam (`invokeGoal(preamble) → yieldSignal`) the same way loom defined `GitRunner`. Then the preflight `claude --version` check (already in the PLAN risk note) becomes a guard the adapter owns, and a unit test can assert the adapter refuses to run on an unpinned/mismatched version — `it('refuses to invoke /goal below the pinned version')` defends against the exact "API evolved underneath us" risk the plan names.

**Parallel-safety — the side-channel is the shared resource, scope it per-invocation.** Vitest runs files in parallel; if dispatch and PR-feedback both write to a fixed side-channel path or module-level state, they race. The loom tests already model the fix: `mkdtempSync` per test, everything scoped under the scratch dir, `CliContext` carries `repoRoot` so nothing reads a global. Carry that exact discipline — the side-channel path must come from injected context, never a hardcoded `/tmp/jelly-dispatch.log` or a module-level array. This is also the idempotency lever for tension (e): if re-running `/jelly-pr-feedback #N` appends duplicate dispatch records, that's a write-once-vs-append bug, and it's unit-testable today — `it('re-invoking feedback on the same PR does not double-dispatch already-fixed comments')` against the injected side-channel. Defer to substrate-engineer on whether the dedup key lives in the loom manifest's write-once posture or in PR/git state, but the *test* for it is a pure assertion on the dispatch-task builder given a prior-state input.

**What I'd leave to CI / defer.** The plugin.json/marketplace dependency declaration (`[commons, jelly-guild, jelly-loom]`) is a manifest-shape check — that's the `three-axis-integrity.test.ts` family pattern, a structural assertion, not a behavior test; don't write behavior tests for it. The turn-based protocol's *visible* rendering (operator turn → substrate turn → agent turn) is presentation; test that the right turn-state transitions fire (a pure state-machine reducer is the natural seam), not the exact prose of the printed turn banner.

For the next round: I'd want substrate-engineer to settle where the idempotency/dedup key lives (manifest vs git/PR state), because that decision picks which fixture each dispatch test depends on — and skeptic to name any failure mode of `/goal` yielding *without* a clean signal (partial yield, error-yield), because each one becomes a `GoalAdapter` unit test or it becomes a dogfood surprise.

Relevant files:

- `plugins/jelly-loom/mcp/server.test.ts` — the `fakeGit`/recorded-calls pattern that generalizes to `fakeDispatcher`/`GoalAdapter`
- `plugins/jelly-guild/three-axis-integrity.test.ts` — the structural-assertion pattern for the plugin.json/dependency shape check
