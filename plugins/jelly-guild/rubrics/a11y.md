# Rubric: a11y

## Scope

Score the artifact's accessibility — whether it works for users
of assistive technology, keyboard-only navigation, reduced
motion, and other non-mouse modalities. Covers semantic markup,
focus management, ARIA usage, keyboard parity, color contrast,
and motion preferences.

The mode file at `plugins/jelly-guild/modes/domains/a11y.md` is
the prose form of this same content. The two are manually
synced in v1; if you find drift, the mode is canonical.

A11y is designed for, not retrofitted. Flag at design seams,
not just at the end.

## Criteria

1. **Interactive controls use semantic elements.** PASS: every
   click-target is `<button>`, `<a href>`, or an appropriate
   form control. FAIL: a `<div>` or `<span>` carries an
   onClick handler, OR an `<a>` is used as a click target
   without `href`, OR `role="button"` is applied to a non-
   button element without full keyboard-handler implementation
   (Enter + Space).

2. **Every interactive control has an accessible name.** PASS:
   each `<button>`, `<a>`, `<input>`, `<select>`, `<textarea>`
   has text content, `aria-label`, `aria-labelledby`, or a
   programmatically-associated `<label>`. FAIL: an icon-only
   button without `aria-label`; a form input without
   associated label or `aria-labelledby`.

3. **Focus moves deliberately on state change.** PASS: after
   modal close, focus returns to the trigger; after route
   change, focus moves to the page heading or main landmark;
   after destructive action, focus moves to a sensible
   adjacent element. FAIL: focus vanishes (lands on
   `<body>`), stays on a now-removed element, or jumps
   unpredictably.

4. **Information uses multiple signal channels, not color
   alone.** PASS: error states use color + icon + text;
   required fields use asterisk + text + ARIA; status uses
   color + icon + label. FAIL: the only visual indicator of
   state is a color change (red border for error, red
   asterisk only, colored dot only).

5. **ARIA attributes match the element's role.** PASS: ARIA
   attributes match the element they sit on; states like
   `aria-expanded`, `aria-pressed`, `aria-busy` track actual
   application state. FAIL: `aria-label` on a non-interactive
   element where screen readers ignore it; `role` that
   contradicts the underlying element's semantic role;
   `aria-hidden="true"` on a focusable element (creates a
   "ghost focus" reachable by keyboard but hidden from
   assistive tech).

6. **Heading hierarchy is sequential.** PASS: heading levels
   follow document outline (`<h1>` per page; `<h2>` for
   sections; `<h3>` for subsections) without skipping levels.
   FAIL: a page jumps from `<h1>` to `<h3>` with no `<h2>`,
   OR heading levels are chosen for visual weight rather
   than structure.

7. **Form inputs have programmatic label associations.**
   PASS: each `<input>`, `<select>`, `<textarea>` has either
   a wrapping `<label>` or a sibling `<label htmlFor="id">`
   referencing the input's `id`. FAIL: visible label text
   sits adjacent to the input without programmatic
   association; placeholder used as the only "label."

8. **Animations respect `prefers-reduced-motion`.** PASS:
   keyframe animations, CSS transitions on transform/opacity
   for emphasis, and JS-driven motion are wrapped in
   `@media (prefers-reduced-motion: reduce)` guards OR
   stopped entirely under the preference. Critical motion
   (focus transitions, state transitions essential to user
   understanding) can remain. FAIL: a new animation runs at
   full speed regardless of `prefers-reduced-motion`.

9. **Modals trap focus while open.** PASS: opening a modal
   moves focus inside the modal; tab cycles within the
   modal's focusable elements; closing returns focus to the
   trigger. FAIL: focus can tab out of an open modal into
   page-background content; modal opens without moving focus
   into it.

10. **No positive `tabindex` values; `tabindex="-1"` used
    correctly.** PASS: `tabindex` is either absent, `0`
    (include in tab order), or `-1` (focusable only by
    script — for programmatic focus management). FAIL:
    positive `tabindex` values reorder the document tab
    sequence, OR `tabindex="-1"` is applied to an
    interactive element that should be reachable by keyboard.

## Severity

- **Blocking** (gate the unit): criteria 1, 2, 4, 5, 7, 9, 10
  when the diff introduces the antipattern. Criterion 3
  blocks for new flows; advisory for inherited gaps the diff
  doesn't touch. Criterion 6 blocks for new pages; advisory
  for inherited skip-patterns. Criterion 8 blocks for
  user-flow-critical UI; advisory for decorative motion.
- **Advisory** (flag but do not gate): any criterion where
  the antipattern existed before the diff and the diff
  merely sits adjacent.

A11y debt compounds; treat new violations as blocking unless
explicitly tagged as advisory for a documented reason.

## Evidence shape

For each finding, cite:

- **Path** to the file and the line(s) carrying the
  antipattern.
- **Pattern name** from the catalog (e.g. "generic element
  used for interaction", "missing accessible name", "ARIA
  attribute on wrong element").
- **Why it fires** in one sentence — the concrete symptom
  (e.g. "this `<div onClick>` is not reachable by keyboard
  and is not announced as interactive by screen readers").
- **Remedy** in one sentence — typically "replace with
  `<button>`", "add `aria-label='...'`", "associate the
  label via `htmlFor`", "guard the keyframe with
  `prefers-reduced-motion`", "trap focus inside the modal."
- For criterion-2 (accessible name) findings: include the
  element's source line and confirm no `aria-label`, no
  `aria-labelledby`, no text content, no associated
  `<label>`.
- For criterion-3 (focus management) findings: name the
  state transition (modal close, route change, async action)
  and where focus ends up vs where it should.

Good evidence is "this control is reachable by mouse but
not by keyboard", "this status uses only color to
distinguish error from success", "this modal opens without
moving focus." Concrete, citeable, fixable.
